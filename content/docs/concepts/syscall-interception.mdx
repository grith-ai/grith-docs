grith intercepts system calls at the OS level before they reach the kernel. This is the mechanism that makes the [CLI supervisor](/docs/concepts/cli-supervisor) possible — it allows grith to wrap any external tool with security enforcement, without modifying the tool itself.

## Platform support

The current launch supports full syscall interception on Linux. macOS and Windows support is coming very soon.

| Platform | Mechanism | Interception level | Status |
|---|---|---|---|
| Linux (x86_64, aarch64) | ptrace + seccomp-BPF | Full individual syscall interception | Supported today |
| macOS (Apple Silicon, Intel) | Endpoint Security based supervisor | Planned | Coming very soon |
| Windows (x86_64) | Minifilter + ETW style supervisor | Planned | Coming very soon |

<Callout type="info" title="Linux provides the strongest guarantees">
For the current launch, every file open, network connect, and process spawn is individually intercepted and evaluated on Linux before the kernel executes it. macOS and Windows will get their own supervised backends shortly after the Linux launch.
</Callout>

## Linux: ptrace + seccomp-BPF

On Linux, grith uses two complementary kernel mechanisms to intercept syscalls with minimal overhead.

### How it works

**seccomp-BPF** installs a Berkeley Packet Filter program into the kernel that runs on every syscall. This BPF program is extremely fast — it executes in kernel space and adds negligible overhead. The filter is configured to trap specific syscall classes (file operations, network operations, process management) while allowing benign syscalls (memory allocation, clock reads) to pass through unimpeded.

**ptrace** catches the trapped syscalls in userspace. When seccomp-BPF traps a syscall, the kernel suspends the calling thread and notifies grith via ptrace. grith reads the syscall number, arguments, and context, then makes an enforcement decision.

The flow for a single syscall:

```
Tool process makes syscall
    ↓
seccomp-BPF filter runs in kernel space
    ↓
Filter traps the syscall (file/network/process)
    ↓
Kernel suspends thread, notifies grith via ptrace
    ↓
grith reads syscall number + arguments
    ↓
Converts to ToolCallType
    ↓
Routes through security proxy → score → ALLOW/QUEUE/DENY
    ↓
ALLOW: ptrace resumes the syscall normally
QUEUE: process frozen (SIGSTOP), awaits human review
DENY: ptrace injects EPERM error, syscall blocked
```

<Callout type="tip" title="Selective trapping keeps overhead low">
seccomp-BPF only traps syscalls that grith needs to evaluate — file opens, network connects, process spawns, and similar security-relevant operations. High-frequency benign syscalls like `clock_gettime`, `mmap`, and `brk` pass through without trapping. This keeps the performance overhead in the low single-digit percent range for most workloads.
</Callout>

### Syscall classes

grith intercepts the following syscall classes on Linux:

| Class | Syscalls | What grith evaluates |
|---|---|---|
| File read | `open`, `openat`, `read`, `readlink`, `stat`, `access` | Path, flags, read target |
| File write | `open` (with write flags), `openat`, `write`, `rename`, `unlink`, `chmod` | Path, content indicators, permissions |
| Network | `connect`, `sendto`, `sendmsg`, `bind`, `listen` | Destination address, port, protocol |
| Process | `execve`, `execveat`, `clone`, `fork`, `vfork` | Command, arguments, environment |
| Dangerous | `ptrace` (from child), `mount`, `chroot`, `pivot_root` | Always high-score or auto-deny |

### Injecting denials

When grith denies a syscall, it does not kill the process. Instead, it uses ptrace to modify the syscall return value to an appropriate error code — typically `EPERM` (permission denied) or `EACCES` (access denied). The tool process receives the error as if the kernel had denied the operation. Well-behaved tools handle these errors gracefully; they see the same error they would get from standard Unix permission enforcement.

## macOS: coming very soon

macOS support is not part of the current Linux launch. The planned implementation uses Apple's Endpoint Security framework so grith can supervise file, process, and network activity with a native platform backend.

## Windows: coming very soon

Windows support is also not part of the current launch. The planned implementation combines file-system interception with Windows-native event streams for process and network visibility.

## Process tree tracking

When grith supervises a tool via `grith exec`, the tool may spawn child processes. A code editor might spawn a language server. A build tool might invoke compilers and linkers. A test runner might spawn application processes.

grith tracks the full process tree of every supervised tool. When a supervised process calls `fork`, `clone`, or `execve`, grith detects the new child process and begins intercepting its syscalls with the same filters and profile applied to the parent.

This prevents a common evasion technique: a malicious tool spawning a child process to perform an action that the parent would not be permitted to do. The child inherits the parent's security profile.

<Terminal command="grith exec --profile claude-code -- claude-code 'run the test suite'" />

If `claude-code` spawns `npm test`, which spawns `node`, which spawns the application under test — every process in that tree is monitored by grith.
