The CLI supervisor is grith's runtime for wrapping external tools with OS-level security. Run `grith exec` in front of any CLI tool and every system call that tool makes is intercepted, scored, and enforced — without modifying the tool itself.

## Basic usage

<Terminal command="grith exec -- claude-code 'fix the authentication bug'" />

This spawns `claude-code` under grith's supervision. Every file read, file write, network connection, and process spawn generated by `claude-code` (and any child processes it creates) passes through the [scoring proxy](/docs/concepts/scoring-proxy) before execution.

To use a specific security profile:

<Terminal command="grith exec --profile claude-code -- claude-code 'refactor the database layer'" />

## Pre-built profiles

grith ships with profiles for popular AI coding tools:

| Profile | Tool | Tuned for |
|---|---|---|
| `claude-code` | Anthropic Claude Code | File editing, terminal commands, project-scoped access |
| `codex` | OpenAI Codex CLI | Code generation, file writes, build commands |
| `aider` | Aider | Git operations, file editing, test execution |
| `generic` | Any CLI tool | Conservative defaults, broad syscall coverage |

Each profile defines the tool's expected behaviour: which paths it should read and write, which network destinations are acceptable, which commands are normal, and which patterns should raise the score.

<ConfigExample title="Excerpt from the claude-code profile" code={`[profile.claude-code]
description = "Anthropic Claude Code - file editing and terminal commands"

allow_read = [
  "\${workdir}/**",
  "/usr/lib/**",
  "/usr/share/**",
]

allow_write = [
  "\${workdir}/src/**",
  "\${workdir}/tests/**",
  "\${workdir}/package.json",
  "\${workdir}/tsconfig.json",
]

deny_read = [
  "~/.ssh/**",
  "~/.aws/**",
  "~/.config/grith/**",
  "~/.gnupg/**",
]

deny_network = [
  "metadata.google.internal",
  "169.254.169.254",
]

allow_commands = [
  "npm *",
  "npx *",
  "node *",
  "git *",
  "tsc *",
]

noise_patterns = [
  "stat /usr/lib/*",
  "openat /proc/self/*",
  "read /dev/urandom",
]
`} />

Profiles are stored in `~/.config/grith/profiles/` and can be customised or extended. See the [Profiles guide](/docs/guides/profiles) for details on creating your own.

## PTY forwarding

Many AI coding tools are interactive — they expect a terminal, display progress indicators, and accept keyboard input. grith supports full PTY (pseudo-terminal) forwarding, so interactive sessions work exactly as they would without supervision:

<Terminal command="grith exec --profile claude-code -- claude-code" />

The tool sees a real terminal. Colours, cursor movement, line editing, and signal handling (Ctrl+C, Ctrl+Z) all work as expected. The developer's experience is identical to running the tool directly, with the addition of security enforcement happening invisibly in the background.

## Process tree tracking

When a supervised tool spawns child processes, grith tracks the entire process tree and applies the same security profile to every descendant. This prevents a tool (or a compromised tool) from evading security by delegating operations to a child process.

For example, when `claude-code` runs `npm test`:

```
grith exec
  └── claude-code           ← supervised
        └── npm test         ← supervised (child)
              └── node       ← supervised (grandchild)
                    └── app  ← supervised (great-grandchild)
```

Every process in this tree has its syscalls intercepted and scored. The profile applied to `claude-code` governs the entire tree.

## Attaching to existing processes

You can attach grith to a process that is already running:

<Terminal command="grith exec --attach <PID>" />

<Callout type="warning" title="Attach limitations">
Attaching to an already-running process means grith did not control the process's initial environment or startup syscalls. Any operations that occurred before attachment are unmonitored. For the strongest security guarantees, always launch tools through `grith exec` from the start.
</Callout>

On Linux, attaching uses `ptrace(PTRACE_ATTACH)` to begin intercepting syscalls from the target process and its children. On macOS, attach provides process lifecycle monitoring (freeze, thaw, terminate) but not individual syscall interception.

## Real-time monitoring

Every supervised session streams events to the grith dashboard via WebSocket. Open the dashboard at `localhost:3141` to see:

- Live feed of tool calls, scores, and decisions
- Filter breakdown for each call
- Process tree visualisation
- Quarantine queue with review actions

The dashboard updates in real time. You can monitor a long-running supervised session in one terminal while the tool runs in another.

<Terminal command="grith dashboard" />

## Noise reduction

In a typical session, a tool might generate thousands of syscalls per second. Many of these are routine operations — reading shared libraries, checking `/proc/self/`, reading `/dev/urandom` — that are safe by definition and would clutter the audit log and consume scoring cycles.

grith supports configurable noise patterns that skip scoring for syscalls matching known-safe patterns:

<ConfigExample title="Noise reduction patterns" code={`[supervisor]
noise_patterns = [
  "stat /usr/lib/*",
  "stat /usr/share/*",
  "openat /proc/self/*",
  "read /dev/urandom",
  "read /dev/null",
  "clock_gettime",
  "mmap ANONYMOUS",
]
`} />

<Callout type="danger" title="Noise patterns bypass scoring entirely">
Syscalls matching noise patterns are not scored, not quarantined, and not logged in the detailed audit trail (they are counted in aggregate statistics). Be conservative with noise patterns. Only add patterns that you are certain are safe regardless of context. When in doubt, let the scoring proxy handle it — the static filters are fast enough that most benign calls score 0.0 and auto-allow in under 1ms.
</Callout>

## Enforcement convergence

The CLI supervisor routes every intercepted syscall through the same [scoring proxy](/docs/concepts/scoring-proxy) used by grith's built-in agent. This means:

- The same 17+ filters evaluate the call
- The same scoring thresholds apply
- The same audit log records the decision
- The same quarantine digest collects ambiguous calls
- The same adaptive learning adjusts from your feedback

There is no "supervisor mode" scoring and "agent mode" scoring. There is one security boundary.

<Callout type="tip" title="The key insight">
`grith exec` makes any CLI tool zero-trust secure without modifying the tool itself. The tool does not need to know about grith, integrate with an API, or change its behaviour in any way. grith operates at the OS level, below the tool's awareness.
</Callout>

## Example session

A typical supervised session looks like this:

<Terminal command="grith exec --profile claude-code -- claude-code 'add input validation to the signup form'" />

Behind the scenes:

1. grith spawns `claude-code` under ptrace (Linux) or process monitoring (macOS).
2. `claude-code` connects to the Anthropic API — grith evaluates the network call, scores it low (expected destination), and allows it.
3. `claude-code` reads project files — grith evaluates each file open, confirms paths are within the work directory, and allows them.
4. `claude-code` writes `src/components/SignupForm.tsx` — grith scores the write, confirms the path is allowed, and allows it.
5. `claude-code` runs `npm test` — grith evaluates the command, confirms it matches the profile's `allow_commands`, and allows it.
6. `npm test` spawns `node` — grith adds it to the process tree and continues monitoring.

The developer sees `claude-code` running normally. Grith's overhead is imperceptible. The audit log records every operation with its score and decision.
