Most AI agent security tools operate on a trust-then-verify model: define an allowlist, approve individual actions when they fall outside it, and hope the operator stays attentive. grith inverts this. Every system call is untrusted by default, evaluated by 17+ independent filters, and scored before execution. There is no implicit trust — not for internal calls, not for previously approved patterns, not for anything.

## Core principles

### Per-call evaluation

Every tool call is evaluated before it executes. There is no warm-up period where calls bypass filters, no batch mode that groups calls together, and no "trusted tool" designation that exempts a tool from evaluation. Call number one and call number ten thousand go through the same pipeline.

<Terminal command="grith exec -- claude-code 'refactor the auth module'" />

Every file read, file write, network connection, and process spawn generated by that session is individually intercepted and scored.

### Defence in depth

grith runs 17+ independent filters across four evaluation phases. No single filter decides the outcome. A tool call that slips past one filter still faces sixteen others, each examining a different dimension of the request — path patterns, argument structure, secret content, network destination, behavioural history, taint propagation, and rate of execution.

<Callout type="info" title="Filter independence">
Filters do not share state during evaluation. Each filter receives the same `ToolCallType` input and produces an independent score. This prevents a compromised or misconfigured filter from influencing others.
</Callout>

### Fail closed

If a filter errors, times out, or returns an unexpected result, the call is denied. There is no fallback to a permissive mode. The system assumes the worst and blocks execution. This applies to every component in the pipeline — if the scoring engine itself encounters an error, the call is denied.

### Minimal privilege

Each tool operates within a profile that defines exactly what it can access. A profile for `claude-code` might allow reads in the project directory and writes to specific output paths, while denying access to `~/.ssh`, `~/.aws`, and anything outside the workspace. The tool never receives more permission than the profile grants.

<ConfigExample title="Excerpt from a supervisor profile" code={`[profile.claude-code]
allow_read = ["\${workdir}/**"]
allow_write = ["\${workdir}/src/**", "\${workdir}/tests/**"]
deny_read = ["~/.ssh/**", "~/.aws/**", "~/.config/grith/**"]
deny_network = ["metadata.google.internal", "169.254.169.254"]
`} />

### Auditable

Every call, every filter result, every score, and every decision is logged. The audit log is append-only and tamper-evident. When a queued call is reviewed by a human, the review decision and reasoning are logged alongside the original call. There are no gaps in the record.

### No implicit trust

Even internal tool calls — calls generated by the built-in agent rather than an external tool — pass through the full security proxy. There is no shortcut for "our own" code. The built-in agent and the CLI supervisor converge on the same enforcement point, as described in the [Architecture](/docs/concepts/architecture) overview.

## How grith differs from allowlist-based tools

Most existing AI agent security tools follow a pattern:

1. Define an allowlist of permitted operations
2. When a call falls outside the allowlist, prompt the user for approval
3. Optionally remember the approval for future calls

This creates two problems:

**Approval fatigue.** In a productive session, an AI agent might generate hundreds of tool calls. If 30% require manual approval, the developer is clicking "allow" every few seconds. Attention degrades. Dangerous calls get approved on autopilot.

**Binary decisions.** A call is either on the list (allowed) or not (blocked/prompted). There is no way to express "this is slightly suspicious but probably fine" versus "this is genuinely dangerous." The security model has no gradient.

grith solves both problems with multi-classifier scoring, inspired by SpamAssassin's approach to email filtering:

| Approach | Allowlist tools | grith |
|---|---|---|
| Evaluation | Binary (allow or prompt) | Weighted composite score (0–20+) |
| Human involvement | Every non-allowlisted call | Only genuine edge cases (5–15%) |
| Filter count | Typically 1–3 rules | 17+ independent filters |
| Adaptability | Manual rule updates | Adaptive weight adjustment from feedback |
| Failure mode | Fail open (user approves everything) | Fail closed (errors = DENY) |

<Callout type="tip" title="The 80-90% auto-allow target">
In a well-tuned deployment, 80–90% of tool calls score below the ALLOW threshold and execute without human intervention. The developer only sees the genuinely ambiguous 5–15% in the quarantine digest. This eliminates approval fatigue while maintaining a strict security posture.
</Callout>

## Trust boundaries

grith enforces three distinct trust boundaries:

1. **Agent to system** — Every tool call from an AI agent (built-in or supervised) passes through the security proxy before reaching the operating system.
2. **Process to children** — When a supervised tool spawns child processes, grith tracks the full process tree and intercepts syscalls from children with the same filters applied to the parent.
3. **Network egress** — Outbound network connections are evaluated for destination, protocol, and content. Connections to cloud metadata endpoints, known exfiltration targets, and unexpected destinations are scored accordingly.

No call crosses any of these boundaries without evaluation.
