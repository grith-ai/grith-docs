`grith` is the single binary that fronts the whole platform. With no arguments it drops
into an interactive REPL; otherwise the first positional is a subcommand.

```
grith <command> [options...]
```

## Global flags

| Flag | What it does |
| --- | --- |
| `--config <path>` | Override the config file location. |
| `--log-level <level>` | One of `trace`, `debug`, `info`, `warn`, `error`. |
| `--no-color` | Disable ANSI colour in output. Useful for piping. |
| `--project <name>` | Override the project name (defaults to the current directory). |

## How to read this reference

- **Commands** are documented one-per-page. Click through from the sidebar.
- **Subcommands** appear on the parent command's page (e.g. `grith canary add`,
  `grith canary remove` are documented under [grith canary](/docs/cli/canary)).
- **Exit codes** are 0 for success, 1 for queue, 2 for deny in proxy-style commands;
  generic non-zero for other failures.

## Where do I start?

Most readers want one of these three:

1. **First-time setup** — run [`grith init`](/docs/cli/init), then point it at an
   agent with [`grith exec`](/docs/cli/exec).
2. **Reviewing decisions** — see [`grith digest`](/docs/cli/digest) and
   [`grith log`](/docs/cli/log).
3. **Operating the daemon** — see [`grith daemon`](/docs/cli/daemon).
