The grith dashboard is a local web interface that visualises security proxy activity, filter performance, cost tracking, and quarantine status in real time. It runs on your machine and reads directly from the local analytics database.

## Starting the dashboard

<Terminal command="grith dashboard start" />

<CodeBlock
  code={`[grith] Dashboard running at http://localhost:3141
[grith] WebSocket connected to proxy
[grith] Press Ctrl+C to stop.`}
  language="text"
/>

The dashboard opens at [http://localhost:3141](http://localhost:3141). It connects to the running proxy via WebSocket, so metrics update live as tool calls flow through the system.

<Callout type="info" title="No proxy required for historical data">
The dashboard can display historical data even when no proxy session is active. Start the dashboard and browse past sessions, audit logs, and aggregated metrics from the SQLite database.
</Callout>

## Decision rate distribution

The primary overview panel shows the proportion of tool calls that resulted in each decision over a configurable time window.

| Metric | Description |
|---|---|
| Allow rate | Percentage of calls that scored below `auto_allow_threshold` |
| Queue rate | Percentage of calls in the quarantine zone |
| Deny rate | Percentage of calls that scored above `auto_deny_threshold` |

The panel includes a time-series chart showing how these rates change over days, weeks, or months. A healthy configuration typically shows 80-90% allow, 5-15% queue, and 1-5% deny. If your deny rate is climbing, investigate which filters are triggering more frequently.

## Filter trigger frequency

A bar chart showing how often each filter contributes a non-zero score. This reveals which filters are doing the most work and which may need tuning.

| Insight | What to look for |
|---|---|
| Dominant filter | One filter contributing to most decisions — may need weight adjustment |
| Silent filter | A filter that never triggers — may be misconfigured or unnecessary |
| Correlated filters | Multiple filters always triggering together — indicates convergent detection |

Click any filter bar to drill down into the specific tool calls it flagged.

## Score histograms

A histogram of composite scores across all evaluated tool calls. The x-axis shows score ranges, and the y-axis shows frequency. The allow and deny thresholds are marked as vertical lines.

The ideal distribution has a clear separation between the allow cluster (near 0) and the deny cluster (above 8.0), with few calls in the queue zone. A bimodal distribution is healthy. A flat or uniform distribution suggests the filters are not discriminating effectively.

## Session timeline

A chronological view of agent sessions with key events annotated:

- **Session start/end** markers
- **Tool call dots** coloured by decision (green for allow, amber for queue, red for deny)
- **Canary trigger markers** for exfiltration detection events
- **Human review markers** showing when a queued item was approved or denied

Hover over any event for details. Click to jump to the full audit record.

## Cost per provider and model

Tracks token usage and estimated cost across LLM providers and models.

| Panel | Description |
|---|---|
| Provider breakdown | Pie chart of spend by provider (Ollama shows $0.00 since it is local) |
| Model breakdown | Bar chart of spend by model within each provider |
| Daily cost trend | Line chart of daily spend with 7-day rolling average |
| Session cost | Per-session cost with tool call count correlation |

<Callout type="tip" title="Ollama is free">
If you are using Ollama for local inference, cost tracking shows zero spend for those sessions. This makes it straightforward to compare the cost of local vs. cloud-hosted models for equivalent tasks.
</Callout>

## Quarantine queue depth

A real-time gauge showing how many items are currently awaiting human review, plus historical trends.

| Metric | Description |
|---|---|
| Current depth | Number of items in the queue right now |
| Oldest item age | How long the oldest unreviewed item has been waiting |
| Inflow rate | How fast new items are being queued (per hour) |
| Drain rate | How fast items are being reviewed (per hour) |

A growing queue depth indicates either a calibration issue (too many calls landing in the queue zone) or a review bottleneck (humans are not reviewing fast enough). Address the former by adjusting thresholds; address the latter by setting up digest notifications.

## Approval rates

Shows the outcomes of human reviews for quarantined tool calls.

| Metric | Description |
|---|---|
| Approval rate | Percentage of queued items that were approved |
| Denial rate | Percentage of queued items that were denied |
| Approval by filter | Which filters' quarantine decisions are most often overridden |
| Feedback loop | How adaptive learning has adjusted filter weights based on review decisions |

A high approval rate (above 80%) suggests the allow threshold is too low — many safe calls are being unnecessarily quarantined. Consider raising `auto_allow_threshold`. A high denial rate suggests the deny threshold could be lowered to auto-deny these calls without human intervention.

<Callout type="warning" title="Do not over-optimise approval rates">
A 100% approval rate is not the goal. Some quarantined calls should genuinely be denied. If you tune thresholds to approve everything, you have effectively disabled the quarantine safety net. Aim for a balance where the queue contains genuinely ambiguous calls that benefit from human judgement.
</Callout>

## Customising the dashboard

The dashboard port and bind address are configurable:

<ConfigExample
  title="Dashboard server settings"
  code={`[server]
dashboard_port = 3141           # change the port
bind_address = "127.0.0.1"     # localhost only (default)
# bind_address = "0.0.0.0"     # expose to local network (use with caution)`}
/>

<Callout type="danger" title="Do not expose the dashboard to the internet">
The dashboard has no authentication. It is designed for local use only. If you need remote access, use an SSH tunnel or a reverse proxy with authentication in front of it. See the [Reverse Proxy Guide](/docs/guides/reverse-proxy) for details.
</Callout>
