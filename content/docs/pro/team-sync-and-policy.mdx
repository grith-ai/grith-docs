<Badge variant="pro">Pro</Badge>

grith Pro's team sync connects individual grith installations to a shared policy layer. Team admins define security policies on the grith.ai dashboard, and team members pull those policies with `grith pro sync`. Optionally, team members can upload anonymised audit summaries to power the team dashboard.

## How sync works

<Terminal command="grith pro sync" />

<CodeBlock
  code={`[grith] Connecting to grith.ai...
[grith] Uploading audit summary (opt-in)... done (142 sessions, 8934 calls)
[grith] Downloading team policies... done (3 policies updated)
[grith] Refreshing license... done (valid until 2027-02-24)
[grith] Sync complete.`}
  language="text"
/>

Each sync performs up to three operations:

1. **Upload audit summary** (opt-in) — Anonymised metrics from your local analytics database
2. **Download team policies** — Security policies defined by your team admin
3. **Refresh license** — Validate and renew your Pro license

### What is uploaded

When audit upload is enabled, grith sends aggregated metrics — not raw audit records. The summary includes:

| Data | Included | Not included |
|---|---|---|
| Session count | Yes | Individual session details |
| Decision distribution (allow/queue/deny counts) | Yes | Specific tool call arguments |
| Filter trigger counts | Yes | What the filters flagged |
| Cost per provider (aggregate) | Yes | Individual API keys or prompts |
| Score distribution histogram | Yes | Per-call scores |

<Callout type="info" title="Opt-in only">
Audit upload is disabled by default. Enable it explicitly in your configuration. No data leaves your machine unless you opt in.
</Callout>

<ConfigExample
  title="Enable audit upload"
  code={`[pro.sync]
upload_audit = true             # default: false
upload_frequency = "on_sync"    # on_sync | daily | weekly`}
/>

### What is downloaded

Team policies are the primary payload of the download step. These are security configurations defined by your team admin on the grith.ai dashboard.

## Policy types

Team admins can define four types of policies:

### Filter thresholds

Override the scoring thresholds for the entire team. This ensures a consistent security baseline.

<CodeBlock
  code={`{
  "policy_type": "filter_thresholds",
  "auto_allow_threshold": 2.0,
  "auto_deny_threshold": 7.0,
  "filter_weights": {
    "secrets_scan": 3.5,
    "network_egress": 2.5,
    "path_traversal": 3.0
  }
}`}
  language="json"
/>

### Allowed and denied paths

Restrict which file paths agents can read from or write to across all team machines.

<CodeBlock
  code={`{
  "policy_type": "path_rules",
  "write_allow": [
    "{{project_root}}/**",
    "/tmp/grith-*/**"
  ],
  "write_deny": [
    "/etc/**",
    "/usr/**",
    "~/.ssh/**",
    "~/.aws/**"
  ],
  "read_deny": [
    "~/.ssh/id_*",
    "~/.aws/credentials"
  ]
}`}
  language="json"
/>

<Callout type="tip" title="Template variables">
Path rules support `{{project_root}}` as a placeholder for the current project directory. This allows portable policies that work across different machines and directory structures.
</Callout>

### Notification routing

Define which notification channels should receive which types of alerts across the team.

<CodeBlock
  code={`{
  "policy_type": "notification_routing",
  "routes": [
    {
      "events": ["deny", "canary_trigger"],
      "channels": ["slack"],
      "slack_channel": "#security-alerts"
    },
    {
      "events": ["queue"],
      "channels": ["desktop"]
    }
  ]
}`}
  language="json"
/>

### Custom filter chains

Share preconfigured filter chains for specific project types.

<CodeBlock
  code={`{
  "policy_type": "filter_chain",
  "name": "web-frontend",
  "description": "Optimised for React/Next.js projects",
  "filters": {
    "write_scope": {"weight": 2.0, "allowed_extensions": [".ts", ".tsx", ".css", ".json"]},
    "network_egress": {"weight": 1.5, "allowed_hosts": ["registry.npmjs.org", "api.github.com"]},
    "command_denylist": {"weight": 3.0, "additional_deny": ["rm -rf /"]}
  }
}`}
  language="json"
/>

## Conflict resolution

When a team policy conflicts with a local configuration, grith applies a deterministic merge strategy:

| Setting type | Winner | Rationale |
|---|---|---|
| Security thresholds | Team policy | Security baselines should be consistent across the team |
| Denied paths | Union of team + local | More restrictive — both deny lists are enforced |
| Allowed paths | Intersection of team + local | More restrictive — a path must be allowed by both |
| Filter weights | Team policy (if specified) | Team admin can enforce minimum weights |
| Notification routing | Team policy (if specified) | Critical alerts must reach the right channels |
| LLM provider | Local config | Personal preference — does not affect security |
| Dashboard port | Local config | Machine-specific setting |
| Log level | Local config | Personal preference |

<Callout type="warning" title="Team policy takes precedence for security">
If your team admin sets `auto_allow_threshold` to 2.0 and you locally set it to 4.0, the team value (2.0) wins. This prevents individual team members from weakening the security posture. The rationale is simple: security settings should be at least as strict as the team baseline.
</Callout>

To see which settings are controlled by team policy:

<Terminal command="grith config --show-source" />

<CodeBlock
  code={`[proxy]
  auto_allow_threshold = 2.0          # team policy (locked)
  auto_deny_threshold = 7.0           # team policy (locked)
  filter_chain = "default"            # local config

[llm]
  default_provider = "anthropic"      # local config

[notifications]
  enabled = true                      # team policy (locked)
  backend = "slack"                   # team policy (locked)`}
  language="text"
/>

Settings marked `(locked)` cannot be overridden locally. Attempting to change them with `grith config set` produces a warning:

<CodeBlock
  code={`[grith] Cannot override proxy.auto_allow_threshold — controlled by team policy.
[grith] Contact your team admin to change this setting.`}
  language="text"
/>

## Managing policies (team admin)

Team admins manage policies from the [grith.ai dashboard](https://grith.ai/dashboard/policies). Changes propagate to team members on their next `grith pro sync`.

Policies can also be managed via the API:

<CodeBlock
  code={`# List active policies
curl -H "Authorization: Bearer $GRITH_PRO_TOKEN" \\
  https://api.grith.ai/v1/team/policies

# Create a new policy
curl -X POST -H "Authorization: Bearer $GRITH_PRO_TOKEN" \\
  -H "Content-Type: application/json" \\
  -d @policy.json \\
  https://api.grith.ai/v1/team/policies`}
  language="bash"
/>

## Sync frequency

By default, sync is manual (`grith pro sync`). For teams that need policies to propagate quickly, configure automatic sync:

<ConfigExample
  title="Automatic sync"
  code={`[pro.sync]
auto_sync = true                # sync on every grith startup
auto_sync_interval = "1h"      # also sync periodically in long sessions
upload_audit = true             # include audit summary in sync`}
/>

<Callout type="info" title="Sync does not block startup">
When `auto_sync` is enabled, grith syncs in the background on startup. The security proxy starts immediately with the last-known policies. If the sync returns updated policies, they take effect on the next tool call evaluation.
</Callout>
