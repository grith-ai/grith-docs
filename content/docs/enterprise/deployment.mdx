grith runs as a local daemon on each developer's machine. In enterprise environments, you need to ensure consistent configuration across a fleet of installations, manage updates without disrupting work, and connect everything back to your security operations infrastructure. This page covers the deployment patterns that make that possible.

<Callout type="info" title="Roadmap">
The centralized management server is planned for grith v2.0. Fleet deployment via configuration management, air-gapped installation, and the patterns described below can be implemented today using standard tooling.
</Callout>

## Deployment architecture

In a team deployment, each developer machine runs its own grith daemon. Policy configuration is distributed from a central source, audit data flows to your SIEM, and updates are managed through your existing infrastructure tooling.

```
                    ┌──────────────────────┐
                    │  Policy Repository   │
                    │  (Git / Config Mgmt) │
                    └──────────┬───────────┘
                               │ policy sync
                 ┌─────────────┼─────────────┐
                 │             │             │
           ┌─────┴─────┐ ┌────┴──────┐ ┌────┴──────┐
           │ Developer  │ │ Developer │ │ Developer │
           │ Machine A  │ │ Machine B │ │ Machine C │
           │            │ │           │ │           │
           │ ┌────────┐ │ │ ┌───────┐ │ │ ┌───────┐ │
           │ │ grith  │ │ │ │ grith │ │ │ │ grith │ │
           │ └───┬────┘ │ │ └──┬────┘ │ │ └──┬────┘ │
           └─────┼──────┘ └────┼──────┘ └────┼──────┘
                 │             │             │
                 └─────────────┼─────────────┘
                               │ audit events
                    ┌──────────┴───────────┐
                    │        SIEM          │
                    └──────────────────────┘
```

## On-premise deployment

grith is a single static binary with no external dependencies. It does not require cloud connectivity to function — all security evaluation happens locally. This makes it straightforward to deploy in on-premise environments.

### System requirements

| Resource | Minimum | Recommended |
|---|---|---|
| CPU | 2 cores | 4 cores |
| RAM | 512 MB | 2 GB |
| Disk | 100 MB (binary + config) | 1 GB (with audit log retention) |
| OS | Linux x86_64/aarch64 | Linux x86_64 or aarch64 |
| Network | None (local operation) | Outbound to LLM provider, SIEM |

<Callout type="info" title="Current launch scope">
Enterprise deployment for the current launch is Linux only. macOS and Windows support is coming very soon.
</Callout>

## Air-gapped environments

grith operates without internet connectivity. In air-gapped deployments, the only difference is how you provide LLM access and distribute updates.

### LLM provider in air-gapped mode

In air-gapped environments, use a locally hosted LLM via Ollama or a similar local inference server:

<ConfigExample code={`[llm]
provider = "ollama"
model = "codestral:latest"
base_url = "http://localhost:11434"

# No external API keys needed
# No outbound network traffic
`} title="Air-gapped LLM configuration" />

<Callout type="tip" title="Supervisor mode works offline">
Even without a local LLM, grith's supervisor mode (`grith exec`) works fully offline. It intercepts syscalls from whatever tool you wrap — no LLM connection required for security enforcement.
</Callout>

### Policy distribution in air-gapped mode

Without network access to a policy server, distribute configuration via your existing file transfer mechanism:

<ConfigExample code={`[enterprise.policy_sync]
enabled = true
source = "file:///opt/grith/policies/current.toml"
watch_filesystem = true
verify_signature = true
signing_key_path = "/etc/grith/policy-signing-key.pub"
`} title="File-based policy sync" />

Transfer updated policies via USB, shared mount, or your organization's approved transfer method. grith watches the filesystem for changes and reloads automatically.

### Update distribution

Transfer the grith binary and packages via your approved media transfer process. Verify integrity before deploying:

<Terminal command="sha256sum -c grith_1.0.0_amd64.deb.sha256" />

grith binaries are signed with a release key. Verify the signature:

<Terminal command="gpg --verify grith_1.0.0_amd64.deb.sig grith_1.0.0_amd64.deb" />

## Fleet deployment

### Ansible

<CodeBlock code={`---
- name: Deploy grith to developer machines
  hosts: developer_machines
  become: true
  vars:
    grith_version: "1.0.0"
    grith_config_source: "files/grith/config.toml"
    grith_policy_source: "files/grith/policy.toml"

  tasks:
    - name: Install grith package
      apt:
        deb: "https://releases.grith.ai/{{ grith_version }}/grith_{{ grith_version }}_amd64.deb"
      when: ansible_os_family == "Debian"

    - name: Install grith package (RHEL)
      yum:
        name: "https://releases.grith.ai/{{ grith_version }}/grith-{{ grith_version }}.x86_64.rpm"
      when: ansible_os_family == "RedHat"

    - name: Deploy base configuration
      template:
        src: "{{ grith_config_source }}"
        dest: /etc/grith/config.toml
        owner: root
        group: grith
        mode: "0640"
      notify: restart grith

    - name: Deploy security policy
      copy:
        src: "{{ grith_policy_source }}"
        dest: /opt/grith/policies/current.toml
        owner: root
        group: grith
        mode: "0640"

    - name: Enable grith service
      systemd:
        name: grith
        enabled: true
        state: started

  handlers:
    - name: restart grith
      systemd:
        name: grith
        state: restarted
`} language="yaml" />

### Puppet

<CodeBlock code={`class grith (
  String $version = '1.0.0',
  String $config_source = 'puppet:///modules/grith/config.toml',
  String $policy_source = 'puppet:///modules/grith/policy.toml',
) {
  package { 'grith':
    ensure => $version,
    source => "https://releases.grith.ai/\${version}/grith_\${version}_amd64.deb",
  }

  file { '/etc/grith/config.toml':
    ensure  => file,
    source  => $config_source,
    owner   => 'root',
    group   => 'grith',
    mode    => '0640',
    require => Package['grith'],
    notify  => Service['grith'],
  }

  file { '/opt/grith/policies/current.toml':
    ensure => file,
    source => $policy_source,
    owner  => 'root',
    group  => 'grith',
    mode   => '0640',
  }

  service { 'grith':
    ensure  => running,
    enable  => true,
    require => [Package['grith'], File['/etc/grith/config.toml']],
  }
}
`} language="puppet" />

### Chef

<CodeBlock code={`# grith/recipes/default.rb

grith_version = node['grith']['version'] || '1.0.0'

remote_file '/tmp/grith.deb' do
  source "https://releases.grith.ai/#{grith_version}/grith_#{grith_version}_amd64.deb"
  checksum node['grith']['checksum']
  action :create
end

dpkg_package 'grith' do
  source '/tmp/grith.deb'
  action :install
end

template '/etc/grith/config.toml' do
  source 'config.toml.erb'
  owner 'root'
  group 'grith'
  mode '0640'
  notifies :restart, 'service[grith]'
end

cookbook_file '/opt/grith/policies/current.toml' do
  source 'policy.toml'
  owner 'root'
  group 'grith'
  mode '0640'
end

service 'grith' do
  action [:enable, :start]
end
`} language="ruby" />

## Update management

### Rolling updates

When deploying updates across a fleet, use a staged rollout to catch issues before they affect the entire organization:

1. **Canary group** (5% of fleet) — Deploy the new version to a small group and monitor for 24 hours
2. **Early adopters** (25%) — Expand to willing teams, monitor for 48 hours
3. **General availability** (100%) — Roll out to the remaining fleet

### Rollback

If a new version introduces issues, rollback to the previous version:

<Tabs items={["Debian/Ubuntu", "RHEL/CentOS"]} storageKey="os-rollback">
<div>

<Terminal command="apt install grith=0.9.0" />

</div>
<div>

<Terminal command="yum downgrade grith-0.9.0" />

</div>
</Tabs>

grith stores its configuration and audit data independently of the binary, so rollbacks do not affect state.

<Callout type="warning" title="Configuration compatibility">
Check the release notes before upgrading. If a new version introduces configuration schema changes, the old binary may not load the new configuration. Rollback the configuration file alongside the binary.
</Callout>

## Network architecture

For team deployments where grith instances need to communicate with shared infrastructure:

| Connection | Direction | Port | Protocol | Required |
|---|---|---|---|---|
| LLM provider (cloud) | Outbound | 443 | HTTPS | If using cloud LLM |
| LLM provider (local) | Loopback | 11434 | HTTP | If using Ollama |
| Policy sync server | Outbound | 443 | HTTPS | If using remote policy sync |
| SIEM / syslog | Outbound | 514/6514 | TCP/TLS | If SIEM export enabled |
| Management server | Outbound | 443 | HTTPS | If centralized management enabled |
| Dashboard | Loopback | 3141 | HTTP/WS | Always (local only by default) |
| SSO callback | Inbound | 3141 | HTTPS | If SSO enabled |

<Callout type="danger" title="Dashboard binding">
By default, the grith dashboard binds to `127.0.0.1` (loopback only). Do not bind it to `0.0.0.0` in production. If remote dashboard access is required, place it behind a reverse proxy with authentication. See the [Reverse Proxy guide](/docs/guides/reverse-proxy) for details.
</Callout>

## High availability

grith runs as a local daemon — there is no central server that constitutes a single point of failure. Each grith instance operates independently. If the daemon stops, the supervised process behaviour depends on your fail-closed configuration:

<ConfigExample code={`[proxy]
fail_mode = "closed"    # "closed" = block all calls if proxy is down
                        # "open" = allow all calls if proxy is down (not recommended)
`} title="Fail mode configuration" />

| Fail mode | Behaviour when daemon is down | Use case |
|---|---|---|
| `closed` | All tool calls are blocked until the daemon restarts | Security-critical environments (recommended) |
| `open` | Tool calls proceed without evaluation | Development environments where uptime matters more |

For the daemon itself, configure systemd to restart on failure:

<CodeBlock code={`[Unit]
Description=grith security daemon
After=network.target

[Service]
Type=simple
ExecStart=/usr/bin/grith daemon
Restart=on-failure
RestartSec=5
WatchdogSec=30

[Install]
WantedBy=multi-user.target
`} language="ini" />
