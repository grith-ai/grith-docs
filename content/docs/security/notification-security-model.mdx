grith's notification system delivers security alerts — denied tool calls, quarantine events, canary triggers — to channels like Slack, email, webhooks, and desktop notifications. Because these alerts contain sensitive operational data, the notification system itself must be secured.

## Secret management

Notification channels require credentials: Slack webhook URLs, SMTP passwords, webhook signing secrets. grith never stores these as plaintext in configuration files. Instead, all secrets use the `*_env` pattern — the configuration value is the name of an environment variable, not the secret itself.

<ConfigExample
  title="Secrets reference environment variables"
  code={`[notifications.slack]
webhook_url_env = "GRITH_SLACK_WEBHOOK"    # references $GRITH_SLACK_WEBHOOK
channel = "#grith-alerts"

[notifications.email]
smtp_password_env = "GRITH_SMTP_PASSWORD"  # references $GRITH_SMTP_PASSWORD
smtp_host = "smtp.example.com"
smtp_port = 587
from = "grith@example.com"
to = ["security-team@example.com"]

[notifications.webhook]
url = "https://your-endpoint.example.com/grith"
secret_env = "GRITH_WEBHOOK_SECRET"        # references $GRITH_WEBHOOK_SECRET`}
/>

<Callout type="danger" title="Never commit secrets">
If you see a plaintext URL containing `hooks.slack.com` or an API key in your `config.toml`, you have a misconfiguration. Always use the `*_env` pattern. grith will warn on startup if it detects what appears to be a plaintext secret in a configuration value that should reference an environment variable.
</Callout>

At runtime, grith reads the environment variable and holds the secret in memory only for the duration of the notification dispatch. Secrets are not written to the audit log, the digest database, or any on-disk cache.

## Channel authentication

Each notification channel uses its own authentication mechanism:

| Channel | Authentication method |
|---|---|
| Slack | Webhook URL (treat as secret, use `webhook_url_env`) |
| Email (SMTP) | SMTP credentials via `smtp_password_env`, TLS required |
| Webhook | HMAC-SHA256 signature on payload body using `secret_env` |
| Desktop | OS-level notification API, no external authentication |

### Webhook signature verification

When grith sends a webhook notification, it computes an HMAC-SHA256 signature over the JSON payload body using the secret from `GRITH_WEBHOOK_SECRET` and includes it in the `X-Grith-Signature` header.

Your receiving endpoint should verify this signature before processing the payload:

<CodeBlock
  code={`import hmac
import hashlib

def verify_signature(payload_body: bytes, signature: str, secret: str) -> bool:
    expected = hmac.new(
        secret.encode(),
        payload_body,
        hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(f"sha256={expected}", signature)`}
  language="python"
/>

<Callout type="warning" title="Use constant-time comparison">
Always use a constant-time comparison function (like `hmac.compare_digest` in Python or `crypto.timingSafeEqual` in Node.js) to prevent timing attacks when verifying signatures.
</Callout>

## Rate limiting

Notification floods are a concern in two scenarios: a misconfigured filter chain that generates excessive alerts, or an attacker deliberately triggering high-volume tool calls to overwhelm the notification channel and cause alert fatigue.

grith applies rate limiting at two levels:

| Level | Default | Configurable |
|---|---|---|
| Per-channel rate limit | 10 notifications per minute | `notifications.rate_limit_per_minute` |
| Global rate limit | 30 notifications per minute across all channels | `notifications.global_rate_limit_per_minute` |

When the rate limit is exceeded, grith batches subsequent notifications into a single summary delivered when the rate window resets. The summary includes the count of suppressed notifications and their severity breakdown.

<ConfigExample
  title="Rate limiting configuration"
  code={`[notifications]
enabled = true
rate_limit_per_minute = 10          # per channel
global_rate_limit_per_minute = 30   # across all channels
batch_suppressed = true             # send summary of suppressed notifications`}
/>

## Routing rules

Not every alert needs to go to every channel. grith supports routing rules that direct notifications based on severity and event type. This ensures critical alerts reach the right people without flooding low-priority channels.

<ConfigExample
  title="Notification routing"
  code={`[[notifications.routes]]
events = ["deny", "canary_trigger"]
severity = ["critical", "high"]
channels = ["slack", "email"]

[[notifications.routes]]
events = ["queue"]
severity = ["medium"]
channels = ["slack"]

[[notifications.routes]]
events = ["queue"]
severity = ["low"]
channels = ["desktop"]`}
/>

With this configuration:

- Denied tool calls and canary triggers at critical or high severity go to both Slack and email
- Medium-severity queued items go to Slack only
- Low-severity queued items show a desktop notification only

If no routing rules are configured, all notifications go to all enabled channels (the default behaviour).

## Audit trail

Every notification dispatch is recorded in the audit log, including:

- Timestamp
- Event type and severity
- Target channel(s)
- Delivery status (sent, failed, rate-limited, batched)
- Error details on failure

This means you can audit not just what happened in the security proxy, but whether the right people were notified about it.

<Terminal command="grith audit --filter event_type=notification" />

## Failure modes

Notification delivery can fail. grith handles failures without blocking the security proxy:

| Failure | Behaviour |
|---|---|
| Channel unreachable | Retry up to 3 times with exponential backoff, then log as failed |
| Invalid credentials | Log error on startup, disable channel, continue with remaining channels |
| Rate limited by provider | Respect provider rate limits, buffer notifications for delayed delivery |
| All channels down | Log locally, continue proxy operation — security decisions are never blocked by notification failures |

<Callout type="info" title="Notifications are advisory, not blocking">
The notification system is a reporting layer. It never blocks or delays security proxy decisions. A denied tool call is denied whether or not the Slack notification succeeds. The audit log is the authoritative record; notifications are a convenience for real-time awareness.
</Callout>
