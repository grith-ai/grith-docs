## Overview

grith is configured via a TOML file at `~/.config/grith/config.toml`. You can generate a default configuration with `grith init` and modify it with `grith config set` or by editing the file directly.

<Terminal command="grith init" />

The configuration is organised into sections that map to grith's major subsystems: general settings, LLM providers, the scoring proxy, adaptive learning, the digest queue, the API server, the supervisor, and notifications.

<Callout type="info" title="Secret management">
Any configuration value that holds a secret (API key, webhook URL, token) supports the `*_env` pattern. Instead of placing the secret directly in the config file, set the value to the name of an environment variable. For example, `api_key_env = "OPENAI_API_KEY"` tells grith to read the key from `$OPENAI_API_KEY` at runtime. This keeps secrets out of version-controlled files.
</Callout>

## [general]

Top-level settings that apply globally.

<ConfigExample title="General settings" code={`[general]
project = "my-project"
log_level = "info"          # trace | debug | info | warn | error
data_dir = "~/.local/share/grith"`} />

| Key | Type | Default | Description |
|---|---|---|---|
| `project` | String | `""` | Active project context. Scopes audit logs and adaptive learning data. |
| `log_level` | Enum | `"info"` | Logging verbosity. One of `trace`, `debug`, `info`, `warn`, `error`. |
| `data_dir` | Path | `~/.local/share/grith` | Directory for audit logs, adaptive weights, and canary data. |

## [llm]

Configure LLM providers for agent tasks. grith supports multiple providers and lets you switch between them without changing your workflow.

<ConfigExample title="LLM provider configuration" code={`[llm]
default_provider = "ollama"

[llm.ollama]
base_url = "http://localhost:11434"
model = "llama3.2"

[llm.openai]
base_url = "https://api.openai.com/v1"
model = "gpt-4o"
api_key_env = "OPENAI_API_KEY"

[llm.anthropic]
base_url = "https://api.anthropic.com"
model = "claude-sonnet-4-20250514"
api_key_env = "ANTHROPIC_API_KEY"`} />

| Key | Type | Default | Description |
|---|---|---|---|
| `default_provider` | String | `"ollama"` | Which provider block to use by default. |

Each provider sub-block supports:

| Key | Type | Description |
|---|---|---|
| `base_url` | URL | API endpoint for the provider. |
| `model` | String | Model identifier. |
| `api_key_env` | String | Environment variable holding the API key. |

<Callout type="tip" title="Local-first by default">
grith defaults to Ollama so that your data never leaves your machine. You can configure cloud providers for tasks that require larger models, and grith will apply the same security controls regardless of provider.
</Callout>

## [proxy]

Configure the scoring proxy that evaluates every tool call.

<ConfigExample title="Proxy configuration" code={`[proxy]
auto_allow_threshold = 3.0
auto_deny_threshold = 8.0
cold_start_calls = 200
review_timeout_seconds = 300`} />

| Key | Type | Default | Description |
|---|---|---|---|
| `auto_allow_threshold` | Float | `3.0` | Tool calls scoring below this are automatically allowed. |
| `auto_deny_threshold` | Float | `8.0` | Tool calls scoring above this are automatically denied. |
| `cold_start_calls` | Integer | `200` | Number of initial calls during which thresholds are widened (QUEUE zone: 2.0-10.0). |
| `review_timeout_seconds` | Integer | `300` | Seconds before a queued tool call times out if not reviewed. |

### [proxy.filters.*]

Per-filter configuration overrides. Each filter can be enabled/disabled and have its weight adjusted.

<ConfigExample title="Per-filter overrides" code={`[proxy.filters.secret_scan]
enabled = true
weight = 1.5

[proxy.filters.path_match]
enabled = true
weight = 1.0
sensitive_paths = [
  "/etc/shadow",
  "/etc/passwd",
  "~/.ssh/",
  "~/.aws/credentials",
]

[proxy.filters.egress_policy]
enabled = true
allowed_domains = ["api.openai.com", "api.anthropic.com"]
denied_domains = ["*.pastebin.com"]`} />

See [Filters Reference](/docs/reference/filters) for the full list of filters and their configuration options.

## [adaptive]

Control the adaptive learning system that adjusts filter weights based on your approve/deny feedback.

<ConfigExample title="Adaptive learning" code={`[adaptive]
enabled = true
learning_rate = 0.1
confidence_threshold = 0.7
min_samples = 10
max_adjustment = 2.0`} />

| Key | Type | Default | Description |
|---|---|---|---|
| `enabled` | Boolean | `true` | Enable adaptive weight adjustment. |
| `learning_rate` | Float | `0.1` | How aggressively weights shift in response to feedback. Lower values are more conservative. |
| `confidence_threshold` | Float | `0.7` | Minimum confidence level required before adaptive adjustments are applied. |
| `min_samples` | Integer | `10` | Minimum number of feedback samples required before adjustments begin. |
| `max_adjustment` | Float | `2.0` | Maximum amount a filter weight can be adjusted up or down from its baseline. |

## [digest]

Configure the digest queue where QUEUE-zone tool calls are held for review.

<ConfigExample title="Digest configuration" code={`[digest]
delivery = "cli"
interval_active = "30m"
interval_idle = "24h"
max_queue_size = 100`} />

| Key | Type | Default | Description |
|---|---|---|---|
| `delivery` | Enum | `"cli"` | Review interface: `cli` for terminal review, `dashboard` for web UI. |
| `interval_active` | Duration | `"30m"` | How often to batch pending items for review notification during active sessions. |
| `interval_idle` | Duration | `"24h"` | How often to deliver digest summaries during idle periods. |
| `max_queue_size` | Integer | `100` | Maximum number of items in the queue before oldest items are auto-denied. |

## [server]

Configure the API and dashboard server.

<ConfigExample title="Server configuration" code={`[server]
host = "127.0.0.1"
port = 3141
dashboard_dir = "dashboard/dist"
cors_origins = ["http://localhost:3000"]

[server.tls]
cert_path = "/path/to/cert.pem"
key_path = "/path/to/key.pem"

[server.rate_limit]
general_rps = 120
write_rps = 30
proxy_test_rps = 10`} />

| Key | Type | Default | Description |
|---|---|---|---|
| `host` | Address | `"127.0.0.1"` | Address the server binds to. |
| `port` | Integer | `3141` | Port for the API and dashboard. |
| `dashboard_dir` | Path | `"dashboard/dist"` | Path to the built dashboard static files, relative to grith's data directory. |
| `cors_origins` | Array | `[]` | Allowed CORS origins. |

### [server.tls]

| Key | Type | Description |
|---|---|---|
| `cert_path` | Path | Path to the TLS certificate file. |
| `key_path` | Path | Path to the TLS private key file. |

### [server.rate_limit]

| Key | Type | Default | Description |
|---|---|---|---|
| `general_rps` | Integer | `120` | Maximum general requests per second per client. |
| `write_rps` | Integer | `30` | Maximum write requests per second per client. |
| `proxy_test_rps` | Integer | `10` | Maximum proxy test requests per second per client. |

<Callout type="warning" title="Bind address">
The server defaults to `127.0.0.1` (localhost only). If you need remote access, see the [Reverse Proxy guide](/docs/guides/reverse-proxy) rather than binding to `0.0.0.0` directly.
</Callout>

## [supervisor]

Configure the syscall-level supervisor for `grith exec`.

<ConfigExample title="Supervisor configuration" code={`[supervisor]
default_profile = "generic"
max_concurrent_sessions = 4
freeze_timeout_seconds = 300

[supervisor.platform]
linux_mechanism = "ptrace"
macos_mechanism = "endpoint-security"
seccomp_pre_filter = false

[supervisor.noise_reduction]
ignore_read_only = true
batch_rapid_reads = true
batch_window_ms = 50`} />

<Callout type="info" title="Launch scope">
The current launch supports Linux only. The platform backend fields below are part of the config surface because macOS and Windows support is coming very soon.
</Callout>

| Key | Type | Default | Description |
|---|---|---|---|
| `default_profile` | String | `"generic"` | Supervisor profile applied when `--profile` is not specified. |
| `max_concurrent_sessions` | Integer | `4` | Maximum number of supervisor sessions that can run simultaneously. |
| `freeze_timeout_seconds` | Integer | `300` | Seconds before a frozen (paused) supervised process is automatically denied and resumed. |

### [supervisor.platform]

| Key | Type | Default | Description |
|---|---|---|---|
| `linux_mechanism` | Enum | `"ptrace"` | Linux interception mode. |
| `macos_mechanism` | Enum | `"endpoint-security"` | macOS supervision backend. |
| `seccomp_pre_filter` | Boolean | `false` | Experimental seccomp pre-filter switch. |

### [supervisor.noise_reduction]

| Key | Type | Default | Description |
|---|---|---|---|
| `ignore_read_only` | Boolean | `true` | Allow pure read-only opens to bypass the proxy unless they hit sensitive-path protections. |
| `batch_rapid_reads` | Boolean | `true` | Coalesce repeated reads to the same file descriptor into a single proxy evaluation. |
| `batch_window_ms` | Integer | `50` | Millisecond window used when batching rapid reads. |

See [Supervisor Profiles](/docs/reference/supervisor-profiles) for profile configuration.

## [notifications]

Configure notification channels for security alerts. Supports multiple channels with severity-based routing.

<ConfigExample title="Notifications overview" code={`[notifications]
enabled = true
default_channel = "desktop"

[notifications.desktop]
enabled = true

[notifications.slack]
enabled = true
webhook_url_env = "GRITH_SLACK_WEBHOOK"
channel = "#security-alerts"

[notifications.email]
enabled = true
smtp_host = "smtp.example.com"
smtp_port = 587
from = "grith@example.com"
to = ["security-team@example.com"]
username_env = "SMTP_USER"
password_env = "SMTP_PASS"`} />

| Key | Type | Default | Description |
|---|---|---|---|
| `enabled` | Boolean | `true` | Master switch for all notifications. |
| `default_channel` | String | `"desktop"` | Channel used when no routing rule matches. |

### Channel blocks

Each channel is configured as a sub-block under `[notifications]`. Available channels:

| Channel | Key fields |
|---|---|
| `desktop` | `enabled` |
| `email` | `enabled`, `smtp_host`, `smtp_port`, `from`, `to`, `username_env`, `password_env` |
| `slack` | `enabled`, `webhook_url_env`, `channel` |
| `telegram` | `enabled`, `bot_token_env`, `chat_id` |
| `discord` | `enabled`, `webhook_url_env` |
| `whatsapp` | `enabled`, `api_key_env`, `phone_number` |
| `teams` | `enabled`, `webhook_url_env` |
| `pagerduty` | `enabled`, `routing_key_env`, `severity_map` |
| `opsgenie` | `enabled`, `api_key_env`, `team` |
| `webhook` | `enabled`, `url_env`, `headers`, `method` |

### [notifications.routing]

Route notifications to different channels based on severity.

<ConfigExample title="Severity-based routing" code={`[notifications.routing]
critical = ["pagerduty", "slack", "email"]
high = ["slack", "email"]
medium = ["slack"]
low = ["desktop"]`} />

### [notifications.rate_limits]

<ConfigExample title="Rate limits" code={`[notifications.rate_limits]
max_per_minute = 10
max_per_hour = 100
cooldown = "30s"`} />

### [notifications.escalation]

<ConfigExample title="Escalation chain" code={`[notifications.escalation]
enabled = true
timeout = "15m"
chain = ["slack", "email", "pagerduty"]`} />

When a critical notification is not acknowledged within the timeout, grith escalates to the next channel in the chain.

See [Notifications](/docs/reference/notifications) for a detailed guide on setting up each channel.
