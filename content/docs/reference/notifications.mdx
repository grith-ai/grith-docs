## Overview

grith's notification system alerts you when the scoring proxy detects high-risk activity. Notifications are routed based on severity, rate-limited to prevent alert fatigue, and can escalate through a chain of channels if not acknowledged.

Eleven channels are supported out of the box: desktop, email, Slack, Telegram, Discord, WhatsApp, Microsoft Teams, PagerDuty, OpsGenie, generic webhooks, and WebSocket (for live dashboard push).

## Quick setup

Enable notifications and configure your first channel:

<ConfigExample title="Minimal notification config" code={`[notifications]
enabled = true
default_channel = "desktop"

[notifications.desktop]
enabled = true`} />

Test it:

<Terminal command="grith notifications test desktop" />

## Severity levels

grith classifies notifications into four severity levels:

| Severity | Triggers | Default colour |
|---|---|---|
| **critical** | Canary token detected, bulk exfiltration, session terminated | Red |
| **high** | Score above deny threshold, secret detected, taint propagation to untrusted sink | Amber |
| **medium** | Score in upper QUEUE zone (6.0-8.0), elevated egress rate, profile violation | Yellow |
| **low** | Score in lower QUEUE zone (3.0-6.0), behavioural deviation, rate limit approach | Blue |

## Channels

### Desktop

Native desktop notifications via the operating system's notification centre.

<ConfigExample title="Desktop notifications" code={`[notifications.desktop]
enabled = true`} />

No additional configuration required. For the current launch, desktop notifications are supported on Linux (via `notify-send` or D-Bus). macOS and Windows notification notes will be published with those platform releases.

### Email

Send alerts via SMTP.

<ConfigExample title="Email notifications" code={`[notifications.email]
enabled = true
smtp_host = "smtp.example.com"
smtp_port = 587
from = "grith@example.com"
to = ["security-team@example.com", "oncall@example.com"]
username_env = "SMTP_USER"
password_env = "SMTP_PASS"`} />

| Key | Type | Description |
|---|---|---|
| `smtp_host` | String | SMTP server hostname |
| `smtp_port` | Integer | SMTP server port (typically 587 for STARTTLS, 465 for SSL) |
| `from` | String | Sender address |
| `to` | Array | Recipient addresses |
| `username_env` | String | Environment variable holding SMTP username |
| `password_env` | String | Environment variable holding SMTP password |

<Callout type="info" title="Secret management">
Use the `*_env` suffix on any field that holds a secret. The value is the name of an environment variable, not the secret itself. This keeps credentials out of your config file.
</Callout>

### Slack

Post alerts to a Slack channel via an incoming webhook.

<ConfigExample title="Slack notifications" code={`[notifications.slack]
enabled = true
webhook_url_env = "GRITH_SLACK_WEBHOOK"
channel = "#security-alerts"`} />

| Key | Type | Description |
|---|---|---|
| `webhook_url_env` | String | Environment variable holding the Slack webhook URL |
| `channel` | String | Target Slack channel (must match webhook configuration) |

To create a webhook, go to your Slack workspace settings, navigate to Incoming Webhooks, and create a new webhook for your target channel. Store the URL in the environment variable specified by `webhook_url_env`.

### Telegram

Send alerts to a Telegram chat or group.

<ConfigExample title="Telegram notifications" code={`[notifications.telegram]
enabled = true
bot_token_env = "GRITH_TELEGRAM_TOKEN"
chat_id = "-1001234567890"`} />

| Key | Type | Description |
|---|---|---|
| `bot_token_env` | String | Environment variable holding the Telegram bot token |
| `chat_id` | String | Telegram chat ID (use negative numbers for groups) |

Create a bot via [@BotFather](https://t.me/BotFather) and add it to your target chat. Use the Telegram API `getUpdates` method to find your chat ID.

### PagerDuty

### Webhook

Generic webhook for integration with any HTTP-based service.

<ConfigExample title="Webhook notifications" code={`[notifications.webhook]
enabled = true
url_env = "GRITH_WEBHOOK_URL"
method = "POST"
headers = { "Authorization" = "Bearer \${GRITH_WEBHOOK_TOKEN}" }`} />

| Key | Type | Description |
|---|---|---|
| `url_env` | String | Environment variable holding the webhook URL |
| `method` | String | HTTP method (default: `POST`) |
| `headers` | Table | Additional HTTP headers. Values can reference environment variables with `$VAR` syntax. |

The webhook payload follows the same JSON format as the WebSocket `filter_trigger` event. See [WebSocket Events](/docs/reference/websocket-events) for the schema.

### WebSocket

Push notifications to connected WebSocket clients in real time. This powers the live dashboard notification feed.

<ConfigExample title="WebSocket notifications" code={`[notifications.websocket]
enabled = true`} />

No additional configuration required. WebSocket notifications are delivered to all clients connected to `ws://localhost:3141/ws/live`. This channel is primarily used by the dashboard for real-time alert display.

## Severity-based routing

Route notifications to different channels based on severity level. This lets you send critical alerts to PagerDuty while routing low-severity events to Slack.

<ConfigExample title="Routing rules" code={`[notifications.routing]
critical = ["pagerduty", "slack", "email"]
high = ["slack", "email"]
medium = ["slack"]
low = ["desktop"]`} />

When a notification's severity matches a routing rule, it is sent to all channels listed for that severity. If no routing rule matches, the `default_channel` is used.

<Callout type="tip" title="Multi-channel routing">
List multiple channels per severity level to ensure critical alerts reach the right people through redundant channels. A canary token detection should hit PagerDuty, Slack, and email simultaneously.
</Callout>

## Rate limiting

Prevent alert fatigue with rate limits on notification delivery.

<ConfigExample title="Rate limiting" code={`[notifications.rate_limits]
max_per_minute = 10
max_per_hour = 100
cooldown = "30s"`} />

| Key | Type | Default | Description |
|---|---|---|---|
| `max_per_minute` | Integer | `10` | Maximum notifications sent per minute across all channels |
| `max_per_hour` | Integer | `100` | Maximum notifications sent per hour |
| `cooldown` | Duration | `"30s"` | Minimum time between notifications of the same type |

When rate limits are exceeded, notifications are buffered and delivered as a digest summary when the rate limit window resets. Critical-severity notifications bypass rate limits.

<Callout type="warning" title="Critical alerts bypass rate limits">
Notifications with `critical` severity are never rate-limited. This ensures canary token detections and bulk exfiltration alerts are always delivered immediately.
</Callout>

## Escalation chains

When critical alerts are not acknowledged, grith can escalate through a chain of notification channels.

<ConfigExample title="Escalation configuration" code={`[notifications.escalation]
enabled = true
timeout = "15m"
chain = ["slack", "email", "pagerduty"]`} />

| Key | Type | Default | Description |
|---|---|---|---|
| `enabled` | Boolean | `false` | Enable escalation chains |
| `timeout` | Duration | `"15m"` | Time to wait for acknowledgement before escalating |
| `chain` | Array | `[]` | Ordered list of channels to escalate through |

The escalation flow:

1. Critical alert fires. grith sends to the first channel in the chain (e.g., Slack).
2. If not acknowledged within `timeout` (15 minutes), grith sends to the second channel (e.g., email).
3. If still not acknowledged, grith sends to the third channel (e.g., PagerDuty).
4. Acknowledgement at any step stops the chain.

Acknowledgement is detected via the digest queue: approving or denying the related digest item counts as acknowledgement.

## Checking notification status

### CLI

<Terminal command="grith notifications status" />

Shows the status of all configured channels, recent notification counts, and rate limit state.

<Terminal command="grith notifications channels" />

Lists all available channels with their configuration state (configured, enabled, connected, error).

### API

<Terminal command="curl http://localhost:3141/notifications/status" />

See the [REST API Reference](/docs/reference/api-rest#notifications) for the full response format.

## Testing channels

Always test channels after configuration to verify connectivity:

<Terminal command="grith notifications test slack" />

This sends a test notification to the specified channel. Check that it arrives correctly before relying on it for production alerts.

<Callout type="tip" title="Test all channels">
Run `grith notifications test <channel>` for each configured channel after setup. A misconfigured channel that silently fails could mean missing a critical alert.
</Callout>
