## Overview

Supervisor profiles define the security policy for `grith exec`. Each profile specifies which file paths, commands, and network destinations a supervised process is allowed to access. When a syscall violates the active profile, grith either blocks it, queues it for review, or logs it, depending on the scoring result.

<Terminal command="grith exec --profile claude-code -- claude" />

grith ships with four pre-built profiles optimised for popular AI coding agents. You can also define custom profiles for any tool.

## How profiles work

When you run `grith exec --profile <name> -- <command>`, grith:

1. Loads the named profile from configuration.
2. Spawns the command under ptrace/seccomp-BPF supervision (Linux) or process-lifecycle monitoring (macOS).
3. Intercepts every syscall the supervised process makes.
4. Evaluates each syscall against the profile rules and the scoring pipeline.
5. Allows, queues, or denies the syscall based on the composite score.

Profile rules feed into the scoring pipeline as additional context. A syscall that matches an `allowed_paths` rule receives a score reduction. A syscall targeting a `denied_paths` entry receives a score increase. The profile does not bypass the scoring pipeline; it adjusts how the pipeline evaluates calls.

<Callout type="warning" title="Generic is restrictive">
The generic profile denies all network access by default (`deny_all_other = true`) and only allows read-only commands. Use a tool-specific profile when one is available, or create a custom profile tailored to your use case.
</Callout>

## Profile configuration

### Path rules

Path rules use glob syntax with two special variables:

| Variable | Expands to |
|---|---|
| `$PROJECT_DIR` | The current working directory when `grith exec` was invoked |
| `$HOME` | The user's home directory |

`allowed_paths` entries reduce the scoring penalty for file access syscalls. `denied_paths` entries increase it. A path not matching either list is evaluated neutrally by the profile (other filters still apply).

### Command rules

Command rules use glob syntax. The `*` wildcard matches any arguments. Commands not in `allowed_commands` are not automatically blocked; they receive a higher score from the profile context but may still pass through the scoring pipeline.

<Callout type="info" title="Profiles adjust scores, not hard blocks">
Profile rules adjust the score that the scoring pipeline computes. A denied path adds points to the composite score, which may push it above the deny threshold. But if you have custom allowlist rules or low weights on the relevant filters, the call could still be allowed. For hard blocks, set the deny threshold low or use the allowlist filter with explicit deny rules.
</Callout>

### Network policy

| Key | Type | Description |
|---|---|---|
| `allowed_domains` | Array | Domains the supervised process is allowed to connect to. Supports glob patterns. |
| `deny_all_other` | Boolean | If `true`, any domain not in `allowed_domains` receives a high score increase. |

### Noise reduction

High-frequency harmless syscalls (like `futex`, `clock_gettime`, `mmap`) generate noise in the supervisor log. The `ignore_patterns` array lists syscall names that should be silently passed through without scoring or logging.

## Custom profiles

Define custom profiles in your configuration file or in `config/supervisor/profiles.toml` in the grith data directory.

<ConfigExample title="Custom profile example" code={`[profiles.my-agent]
description = "Custom profile for my internal agent"

allowed_paths = [
  "$PROJECT_DIR/**",
  "$HOME/.config/my-agent/**",
  "/var/data/my-agent/**",
]

denied_paths = [
  "$HOME/.ssh/**",
  "$HOME/.aws/**",
  "/etc/**",
]

allowed_commands = [
  "git *",
  "python *",
  "pip *",
  "docker *",
  "kubectl *",
]

[profiles.my-agent.network_policy]
allowed_domains = [
  "api.openai.com",
  "*.internal.company.com",
  "registry.npmjs.org",
]
deny_all_other = true

[profiles.my-agent.noise_reduction]
ignore_patterns = [
  "futex",
  "clock_gettime",
  "epoll_wait",
  "mmap",
  "munmap",
  "mprotect",
  "brk",
]`} />

<Terminal command="grith exec --profile my-agent -- python my_agent.py" />

## Platform support

| Platform | Interception level | Notes |
|---|---|---|
| Linux x86_64 / aarch64 | Full syscall (ptrace + seccomp-BPF) | Recommended. All profile rules enforced at syscall level. |
| macOS (Apple Silicon / Intel) | Process lifecycle | File and network access monitored at process boundaries. Less granular than Linux. |
| Windows | Planned | Not yet supported. See roadmap. |

<Callout type="tip" title="Linux recommended for supervisor">
For the most effective supervision, run `grith exec` on Linux where ptrace and seccomp-BPF provide true syscall-level interception. On macOS, grith monitors process lifecycle events which covers file creation, network connections, and process spawning but cannot intercept individual read/write operations within a process.
</Callout>
