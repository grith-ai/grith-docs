## Overview

Supervisor profiles describe the routine behavior grith should expect from a supervised tool: which paths it normally touches, which commands it spawns, which network destinations it contacts, and which launch arguments are required. grith uses those entries to build a low-score session allowlist for `grith exec`.

<Terminal command='grith exec --profile claude-code -- claude "review the auth changes"' />

Profiles do not replace the [scoring proxy](/docs/concepts/scoring-proxy). They narrow what is treated as routine. Matching entries lower scores and seed fast-path allowlist entries; non-matching operations still go through the full filter pipeline.

## Resolution model

Every supervised session resolves to one effective policy:

1. `[defaults]` adds universal local-only trust shared by every profile.
2. The selected static profile is resolved through `extends` inheritance (`child -> parent -> [defaults]`).
3. A provider overlay may add destinations for `grith-repl` sessions.
4. A launcher overlay may add small context-specific trust for IDE terminals.

<Callout type="info" title="No launcher or provider flags on grith exec">
`grith exec` exposes `--profile`, not `--launcher` or `--provider`. Launcher overlays are detected automatically at runtime. Provider overlays are used by grith's built-in agent and REPL flows, not by external tool supervision.
</Callout>

## Profile selection

When you run `grith exec`, grith chooses the base profile in this order:

1. `--profile <name>` if you provide one
2. Auto-detection from the command name
3. `supervisor.default_profile` from config

Auto-detection currently recognizes these commands:

| Command | Selected profile |
|---|---|
| `claude`, `claude-code` | `claude-code` |
| `codex` | `codex` |
| `aider` | `aider` |
| `goose` | `goose` |
| `copilot`, `copilot-cli` | `copilot` |
| `cursor-agent` | `cursor` |
| `cline` | `cline` |
| `openclaw` | `openclaw` |

Unknown tools fall back to the configured default profile, which ships as `generic`.

## Shipped profiles

grith currently ships with 11 tool/base profiles:

| Profile | Role | Notes |
|---|---|---|
| `generic` | Strict fallback | Project-local trust only; default fallback |
| `generic-cli` | Manual opt-in | Broader shared developer CLI trust for generic tools |
| `grith-repl` | Built-in agent base | Used with provider overlays for REPL and `grith run` tool execution |
| `claude-code` | Claude Code | Includes launch contract and Anthropic/GitHub/VS Code trust |
| `codex` | OpenAI Codex CLI | OpenAI plus common toolchain trust |
| `aider` | Aider | Aider state, git, and model-provider destinations |
| `goose` | Goose | Goose workspace/state plus provider destinations |
| `copilot` | GitHub Copilot CLI | GitHub/Microsoft endpoints and common local tooling |
| `cursor` | Cursor CLI | Includes launch contract `--sandbox disabled` |
| `cline` | Cline | Cline extension/runtime destinations |
| `openclaw` | OpenClaw | Standalone profile for browser, messaging, and provider tooling |

The shipped definitions live in `config/supervisor/profiles.toml`.

## Launcher overlays

Launcher overlays are additive-only runtime layers. They exist to reflect how the session was launched, without creating a combinatorial explosion of profile variants.

| Overlay | Detection | Additions |
|---|---|---|
| `vscode-terminal` | Parent process is `code`, `code-insiders`, or `codium`; or `TERM_PROGRAM=vscode` | Adds `code` to routine commands |
| `cursor-terminal` | Parent process is `cursor` | Placeholder overlay for Cursor-launched terminals |

Launcher overlays cannot remove trust or override the base profile. They only add a small amount of low-risk context.

## Provider overlays

Provider overlays add network destinations for grith's built-in agent flows. They apply to `grith-repl`, not to named external-tool profiles.

| Overlay | Destinations added |
|---|---|
| `openai` | `openai.com`, `api.openai.com`, `auth.openai.com`, `chatgpt.com` |
| `anthropic` | `anthropic.com`, `api.anthropic.com`, `claude.ai`, `claude.com` |
| `openrouter` | `openrouter.ai` |

## Profile schema

Profiles are defined as `[[profiles]]` entries in `config/supervisor/profiles.toml`.

<ConfigExample
  title="Custom profile example"
  code={`[[profiles]]
name = "my-tool"
display_name = "My Tool"
extends = "generic"
rationale = "My Tool edits project files, runs tests, and calls its API."

routine_paths = [
    "\${PROJECT_DIR}/**",
    "\${HOME}/.config/my-tool/**",
    "\${HOME}/.cache/my-tool/**",
    "/tmp/my-tool-*",
]

routine_commands = [
    "python", "python3", "pytest",
    "cargo", "rustc",
    "curl",
]

routine_destinations = [
    "api.my-tool.com",
    "pypi.org",
    "files.pythonhosted.org",
]

readonly_paths = [
    "\${HOME}/.config/my-tool/credentials.json",
]

readonly_path_patterns = [
    "\${HOME}/.ssh/*.pub",
]

routine_exec_roots = []
routine_listen_addresses = []`}
/>

### Fields

| Field | Type | Purpose |
|---|---|---|
| `name` | string | Machine-readable identifier used with `--profile` |
| `display_name` | string | Human-readable name for logs and UI |
| `extends` | string | Optional parent profile name |
| `rationale` | string | Short explanation of the profile's trust model |
| `routine_paths` | string[] | Paths trusted for routine file I/O |
| `routine_commands` | string[] | Executable basenames trusted for routine process spawns |
| `routine_destinations` | string[] | Hostnames trusted for routine outbound network access |
| `routine_exec_roots` | string[] | Trusted executable root directories for spawn allowlisting |
| `readonly_paths` | string[] | Exact-match read-only paths |
| `readonly_path_patterns` | string[] | Simple `*`-wildcard read-only patterns |
| `routine_listen_addresses` | string[] | Addresses the tool may bind without prompting |
| `launch_contract.required_args` | string[] | Args grith must inject or enforce before launch |

## Matching rules

### routine_paths

`routine_paths` are recursive trust entries. A path like `${PROJECT_DIR}/**` allows reads and writes anywhere under the current project root. A path like `/tmp/my-tool-*` trusts matching temp directories.

Use these carefully: `routine_paths` widen trust for both reads and writes.

### readonly_paths

`readonly_paths` trust reads only and use exact matching. A trusted entry for `${HOME}/.ssh/config` does not also trust `${HOME}/.ssh/config.bak`.

### readonly_path_patterns

`readonly_path_patterns` support a narrow `*` wildcard for variable filenames such as public SSH keys. They still apply to reads only.

### routine_commands

`routine_commands` are matched by executable basename. At session start, grith resolves those names on `PATH` and stores the resulting absolute executable paths in the session allowlist.

### routine_destinations

`routine_destinations` use subdomain matching. Adding `anthropic.com` covers `api.anthropic.com` and other subdomains when the event handler evaluates network destinations.

### routine_exec_roots

`routine_exec_roots` trust executable directories, not general file I/O. They seed `exec-prefix:` entries for process-spawn matching after provenance checks.

## Launch contracts

Some tool profiles require specific launch arguments to avoid conflicting with grith's own supervision model.

| Profile | Required args |
|---|---|
| `claude-code` | `--dangerously-skip-permissions` |
| `cursor` | `--sandbox disabled` |

grith enforces those launch contracts before spawning the child process. If the active command line conflicts with the profile contract, launch fails instead of silently proceeding.

## Session allowlist construction

After resolving the effective policy, grith flattens it into a session allowlist used by the syscall event handler.

| Source | Resulting entry type | Match style |
|---|---|---|
| `routine_paths` | path entry | Prefix / recursive |
| `routine_commands` | resolved executable path | Exact executable path |
| `routine_destinations` | `net:` entry | Subdomain-aware hostname match |
| `readonly_paths` | `ro:` entry | Exact match, reads only |
| `readonly_path_patterns` | `ro-glob:` entry | Simple wildcard, reads only |
| `routine_exec_roots` | `exec-prefix:` entry | Executable path prefix |
| `routine_listen_addresses` | listen-address entry | Exact address match |

That allowlist is then combined with the normal proxy filters, reputation, taint tracking, rate limits, and audit logging.

## Defaults

`[defaults]` is merged into every profile. The shipped defaults are intentionally local-only:

- common shell and file-inspection commands such as `bash`, `sh`, `git`, `ls`, `grep`, `rg`, and `ssh`
- system introspection paths like `/proc` and `/etc/profile.d`
- git helper executables under `/usr/lib/git-core/`
- read-only SSH and git config trust for `~/.ssh/config`, `known_hosts`, and `~/.gitconfig`

There are no shared network destinations in `[defaults]`.
