## Overview

Every tool call that passes through grith's proxy receives a composite risk score. This score determines the call's fate: allowed immediately, queued for human review, or denied outright. The scoring system is designed to be transparent, tunable, and adaptive.

## Score zones

The composite score maps to one of three decision zones:

| Zone | Score range | Action |
|---|---|---|
| **ALLOW** | Below 3.0 | Tool call executes immediately |
| **QUEUE** | 3.0 to 8.0 | Tool call is held for human review in the digest |
| **DENY** | Above 8.0 | Tool call is blocked and logged |

The thresholds are configurable via `auto_allow_threshold` and `auto_deny_threshold` in the `[proxy]` configuration section.

<ConfigExample title="Custom thresholds" code={`[proxy]
auto_allow_threshold = 3.0
auto_deny_threshold = 8.0`} />

<Callout type="info" title="Threshold tuning">
Lowering `auto_allow_threshold` makes grith more cautious (more calls go to QUEUE). Raising `auto_deny_threshold` makes it more permissive (fewer hard denials). Adjust based on your risk tolerance and the frequency of false positives.
</Callout>

## Cold-start behaviour

During the first `cold_start_calls` tool calls (default: 200), grith widens the QUEUE zone to capture more calls for review. This lets the adaptive learning system gather feedback before narrowing to normal thresholds.

| Parameter | Cold-start value | Normal value |
|---|---|---|
| ALLOW threshold | 2.0 | 3.0 (configurable) |
| DENY threshold | 10.0 | 8.0 (configurable) |

<ConfigExample title="Cold-start configuration" code={`[proxy]
cold_start_calls = 200`} />

During cold start, the QUEUE zone spans 2.0 to 10.0, capturing a wider band of tool calls for human review. As you approve and deny calls, the adaptive system learns your preferences. After 200 calls, thresholds snap to their configured values.

## Composite score computation

The composite score is a weighted sum of individual filter scores:

```
composite = sum(filter_score[i] * weight[i]) for all enabled filters
```

Each filter produces a raw score (see [Filters Reference](/docs/reference/filters) for per-filter score ranges). The raw score is multiplied by the filter's weight before being added to the composite.

### Worked example

Consider a tool call to `write_file` with path `/etc/hosts`:

| Filter | Raw score | Weight | Weighted score |
|---|---|---|---|
| `operation_risk` | +2 (write) | 1.0 | 2.0 |
| `path_match` | +3 (system config) | 1.0 | 3.0 |
| `sensitive_path` | +4 (system config) | 1.0 | 4.0 |
| `allowlist` | 0 (no match) | 1.0 | 0.0 |
| `argument` | 0 (normal) | 1.0 | 0.0 |
| `behavioural` | +2 (deviation) | 1.0 | 2.0 |
| **Composite** | | | **11.0** |

With the default `auto_deny_threshold` of 8.0, this call is denied. The audit log records the composite score and every filter's individual contribution for full transparency.

<Callout type="danger" title="Deny: composite score 11.0">
Writing to `/etc/hosts` triggers multiple filters. The `path_match` and `sensitive_path` filters alone push the score past the deny threshold. This is by design: operations on system-critical files should require explicit allowlisting.
</Callout>

### Another example

A tool call to `read_file` with path `/home/user/project/src/main.rs`:

| Filter | Raw score | Weight | Weighted score |
|---|---|---|---|
| `operation_risk` | +1 (read) | 1.0 | 1.0 |
| `path_match` | 0 (no match) | 1.0 | 0.0 |
| `sensitive_path` | 0 (no match) | 1.0 | 0.0 |
| `allowlist` | -1 (matches allow rule) | 1.0 | -1.0 |
| `argument` | 0 (normal) | 1.0 | 0.0 |
| `behavioural` | 0 (consistent) | 1.0 | 0.0 |
| **Composite** | | | **0.0** |

Score of 0.0 falls in the ALLOW zone. The call proceeds immediately.

## Filter weights

Every filter has a configurable weight that scales its contribution to the composite score. The default weight is 1.0.

<ConfigExample title="Adjusting filter weights" code={`[proxy.filters.secret_scan]
weight = 1.5    # Increase secret detection influence

[proxy.filters.argument]
weight = 0.5    # Reduce argument analysis influence`} />

Weight adjustments are useful when you want to emphasise or de-emphasise specific filters without disabling them entirely. A weight of 0.0 effectively silences a filter without removing its audit trail.

## Adaptive learning

When adaptive learning is enabled, grith adjusts filter weights automatically based on your approve/deny feedback from the digest queue.

### How it works

1. A tool call lands in the QUEUE zone.
2. You review the call and approve or deny it.
3. grith records which filters contributed to the score.
4. If you approved and the score was high (false positive), the weights of the contributing filters are decreased slightly.
5. If you denied and the score was low (false negative), the weights of the contributing filters are increased slightly.

The adjustment magnitude is controlled by `learning_rate` (default: 0.1). Adjustments only apply once `min_samples` feedback signals have been collected and the confidence level exceeds `confidence_threshold`. No single filter weight can be adjusted by more than `max_adjustment` from its baseline.

<ConfigExample title="Adaptive learning configuration" code={`[adaptive]
enabled = true
learning_rate = 0.1
confidence_threshold = 0.7
min_samples = 10
max_adjustment = 2.0`} />

<Callout type="tip" title="Bounded convergence">
With the default learning rate of 0.1 and a maximum adjustment of +/-2.0, the adaptive system converges meaningfully while preventing any single filter from being amplified or silenced beyond safe bounds. At least 10 feedback samples are required before adjustments begin.
</Callout>

### Viewing adaptive state

You can inspect current adaptive weights and feedback statistics via the CLI or API:

<Terminal command="grith config" />

The adaptive weights are persisted in `data_dir` and survive restarts.

## Early termination

If the running composite score exceeds `auto_deny_threshold` at any phase boundary (after Static, after Pattern, or after Context), remaining phases are skipped. The call is denied immediately with the current score.

This means a call that triggers `path_match` (+5) and `sensitive_path` (+4) in the Static phase alone will be denied without running Pattern or Context filters, keeping latency low for obviously dangerous calls.

## Score in audit logs

Every tool call's score is recorded in the audit log with full detail:

- Composite score
- Per-filter raw scores and weighted scores
- Decision (allow, queue, deny)
- Phase at which early termination occurred (if applicable)
- Adaptive weight snapshot at time of evaluation

<Terminal command="grith audit" />

Use `grith audit export --format json` to export scores for analysis in external tools.

## Tuning recommendations

| Goal | Adjustment |
|---|---|
| Fewer false positives (too many QUEUEs) | Raise `auto_allow_threshold`, lower weights on noisy filters |
| Fewer false negatives (dangerous calls allowed) | Lower `auto_allow_threshold`, increase weights on security-critical filters |
| Faster convergence during cold start | Reduce `cold_start_calls` (but allow enough calls for meaningful feedback) |
| More conservative stance | Lower both thresholds, increase `secret_scan` and `taint` weights |
| More permissive stance | Raise both thresholds, add project paths to allowlist |

<Callout type="warning" title="Test before deploying">
Always use `grith proxy test` to validate threshold and weight changes before applying them to a live session. Aggressive changes can cause either excessive blocking or insufficient protection.
</Callout>
