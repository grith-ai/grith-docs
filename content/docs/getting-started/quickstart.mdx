## Prerequisites

Before starting, make sure you have grith installed and initialised. If not, follow the [Installation](/docs/getting-started/installation) guide first.

<Terminal command="grith --version" />

## Initialise grith

If you have not already run `grith init`, do so now. This creates your configuration file at `~/.config/grith/config.toml` with sensible defaults.

<Terminal command="grith init" />

You should see output confirming the config was created:

<CodeBlock
  code={`✓ Created ~/.config/grith/config.toml
✓ Default filter chain loaded (17 filters)
✓ Digest database initialised at ~/.local/share/grith/digest.db
✓ Ready. Run 'grith run' to start your first task.`}
  language="text"
/>

## Configure an LLM provider

grith needs an LLM provider for its built-in agent. Ollama is the default and runs entirely on your machine — no API keys required.

<Tabs items={["Ollama (default)", "OpenAI", "Anthropic"]} storageKey="quickstart-provider">
<div>

Install Ollama from [ollama.com](https://ollama.com), then pull a model:

<Terminal command="ollama pull llama3.1:8b" />

No additional configuration needed. grith connects to Ollama at `http://localhost:11434` by default.

</div>
<div>

Set your API key via an environment variable:

<Terminal command="export OPENAI_API_KEY=sk-..." />

Or set it directly in your config file:

<ConfigExample
  title="~/.config/grith/config.toml"
  code={`[llm.openai]
api_key = "sk-..."`}
/>

Then set OpenAI as the default provider:

<Terminal command="grith config set llm.default_provider openai" />

</div>
<div>

Set your API key via an environment variable:

<Terminal command="export ANTHROPIC_API_KEY=sk-ant-..." />

Or set it directly in your config file:

<ConfigExample
  title="~/.config/grith/config.toml"
  code={`[llm.anthropic]
api_key = "sk-ant-..."`}
/>

Then set Anthropic as the default provider:

<Terminal command="grith config set llm.default_provider anthropic" />

</div>
</Tabs>

<Callout type="info" title="Provider details">
See [LLM Providers](/docs/getting-started/providers) for the full list of supported providers and configuration options.
</Callout>

## Run your first task

Give grith a simple task. Every tool call the agent makes will pass through the security proxy where 17+ filters evaluate it before execution.

<Terminal command='grith run "create a hello world script in Python"' />

grith starts its agent, intercepts every system call, and evaluates each one against the filter chain. You will see output like this:

<CodeBlock
  code={`[grith] Starting agent session...
[grith] Task: create a hello world script in Python

[proxy] Tool call: fs.write → hello.py
  ├─ path_traversal       0.0  PASS
  ├─ extension_allowlist  0.0  PASS
  ├─ write_scope          0.0  PASS
  ├─ secrets_scan         0.0  PASS
  ├─ entropy_check        0.0  PASS
  └─ aggregate score      0.0

[proxy] Decision: ALLOW (score 0.0 < threshold 3.0)
[agent] ✓ Wrote hello.py

[proxy] Tool call: process.exec → python hello.py
  ├─ command_denylist     0.0  PASS
  ├─ network_egress       0.0  PASS
  ├─ resource_limits      0.0  PASS
  └─ aggregate score      0.0

[proxy] Decision: ALLOW (score 0.0 < threshold 3.0)
[agent] ✓ Executed: Hello, world!

[grith] Session complete. 2 tool calls, 2 allowed, 0 queued, 0 denied.`}
  language="text"
/>

Every tool call gets a numeric score. The proxy compares that score against three thresholds:

| Score range | Decision | What happens |
|---|---|---|
| Below `auto_allow_threshold` (default 3.0) | **ALLOW** | Executed immediately |
| Between allow and deny thresholds | **QUEUE** | Held for human review |
| Above `auto_deny_threshold` (default 8.0) | **DENY** | Blocked outright |

## Review queued items

When a tool call scores between the allow and deny thresholds, grith queues it for human review. Use the digest to see what is waiting:

<Terminal command="grith digest review" />

<CodeBlock
  code={`Queued items (2):

 #  Score  Tool call                        Reason
 1  4.2    process.exec → curl example.com  network_egress flagged external request
 2  3.8    fs.write → /etc/hosts            write_scope flagged system path

[a]pprove  [d]eny  [s]kip  [q]uit`}
  language="text"
/>

You can approve or deny each item individually. Approved items execute immediately; denied items are discarded.

<Callout type="tip" title="Adjusting thresholds">
If too many safe operations are being queued, lower the allow threshold with `grith config set proxy.auto_allow_threshold 3.0`. See [Configuration](/docs/getting-started/configuration) for details.
</Callout>

## Wrap external CLI tools

grith is not limited to its built-in agent. You can wrap any external AI tool with `grith exec`, intercepting all its system calls through the same security proxy.

<Terminal command='grith exec -- claude-code "fix the bug in src/main.rs"' />

Everything the wrapped tool does — file reads, writes, network calls, process spawns — passes through grith's filter chain. The external tool does not need to know about grith; interception happens at the syscall level.

<CodeBlock
  code={`[grith] Wrapping: claude-code "fix the bug in src/main.rs"
[grith] Intercepting syscalls via ptrace + seccomp-BPF

[proxy] Tool call: fs.read → src/main.rs
  └─ aggregate score 0.0 → ALLOW

[proxy] Tool call: fs.write → src/main.rs
  ├─ write_scope    0.0  PASS
  ├─ secrets_scan   0.0  PASS
  └─ aggregate score 0.0 → ALLOW

[grith] Session complete. 2 tool calls, 2 allowed, 0 queued, 0 denied.`}
  language="text"
/>

<Callout type="warning" title="Linux required for full interception">
The current launch supports Linux only. macOS and Windows support is coming very soon, but this quickstart assumes a Linux environment today.
</Callout>

## Launch the dashboard

grith includes a web dashboard for real-time visibility into agent activity, filter results, and session history.

<Terminal command="grith dashboard start" />

<CodeBlock
  code={`[grith] Dashboard running at http://localhost:3141
[grith] Press Ctrl+C to stop.`}
  language="text"
/>

Open [http://localhost:3141](http://localhost:3141) in your browser. The dashboard shows:

- **Live session view** — Tool calls flowing through the proxy in real time
- **Filter results** — Which filters fired, their scores, and the final decision
- **Session history** — Past sessions with full audit trails
- **Digest queue** — Queued items awaiting review
