## Overview

When you wrap an external CLI tool with `grith exec`, the supervisor intercepts every syscall that tool makes and routes it through the security proxy. Supervisor profiles define per-tool security policies: which paths the tool can access, which commands it can run, what network access it gets, and what noise to filter out.

grith ships with pre-built profiles for popular AI tools and supports fully custom profiles for anything else.

## Pre-built profiles

grith includes profiles for the most common AI coding tools:

| Profile | Tool | Optimised for |
|---|---|---|
| `claude-code` | Claude Code (Anthropic) | File edits, shell commands, search operations |
| `codex` | Codex CLI (OpenAI) | Container-style sandboxed execution |
| `aider` | Aider | Git operations, file edits, test runs |
| `generic` | Any CLI tool | Conservative defaults, broad compatibility |

Use a pre-built profile with the `--profile` flag:

<Terminal command='grith exec --profile claude-code -- claude-code "fix the bug in src/main.rs"' />

<Terminal command='grith exec --profile aider -- aider --model ollama/llama3.1' />

If you omit `--profile`, grith uses the `generic` profile.

## Profile fields

Profiles are defined in TOML. Each profile is a section in `~/.config/grith/supervisor/profiles.toml` (or in the system-wide config at `/etc/grith/supervisor/profiles.toml`).

Here is the full schema with all available fields:

<ConfigExample
  title="~/.config/grith/supervisor/profiles.toml"
  code={`[my-profile]
description = "Custom profile for my-tool"

# Filesystem access control
routine_paths = [
    "~/projects/",        # Recursive access to projects directory
    "/tmp/",              # Temp directory access
]
denied_paths = [
    "~/.ssh/",            # Block SSH key access
    "~/.gnupg/",          # Block GPG key access
    "~/.aws/",            # Block AWS credentials
    "/etc/shadow",        # Block shadow password file
]

# Command execution control
routine_commands = [
    "git",
    "npm",
    "node",
    "python",
    "pytest",
    "cargo",
    "make",
]

# Network destinations
routine_destinations = [
    "api.openai.com",
    "api.anthropic.com",
    "registry.npmjs.org",
]

# Network policy
# Options: "allow", "deny", "project-only", "score"
network_policy = "score"

# Noise reduction: suppress known-safe repeated syscalls
[my-profile.noise_reduction]
enabled = true
deduplicate_reads = true           # Collapse repeated reads to same path
suppress_stat_calls = true         # Suppress high-frequency stat/lstat
suppress_cwd_lookups = true        # Suppress getcwd() noise
max_events_per_path_per_second = 5 # Rate-limit events per path`}
/>

### Field reference

#### routine_paths

A list of filesystem paths the tool routinely accesses. Paths ending with `/` grant recursive access to the directory and all children. Paths support `~` expansion for the home directory.

File access within `routine_paths` receives a score reduction. File access outside these paths is evaluated normally by the scoring pipeline. Combined with the deny threshold, this effectively controls what the tool can access.

#### denied_paths

A list of filesystem paths the tool is explicitly blocked from accessing. Denied paths take precedence over routine paths. Use this to carve out exclusions within a routine directory.

#### routine_commands

A list of command names the tool routinely executes. Commands in this list receive a score reduction when evaluated by the scoring pipeline.

Command names are matched against the base name of the executable, not the full path. Arguments are not matched — use other filters for argument inspection.

#### routine_destinations

A list of network destinations (hostnames) the tool routinely connects to. Connections to these destinations receive a score reduction. Connections to destinations not in this list are evaluated normally by the scoring pipeline.

#### network_policy

Controls how network operations are handled:

| Value | Effect |
|---|---|
| `"allow"` | All network access permitted (still logged) |
| `"deny"` | All network access blocked |
| `"project-only"` | Only connections to configured project endpoints allowed |
| `"score"` | Network calls pass through the full filter chain (default) |

#### noise_reduction

The supervisor intercepts syscalls at a very low level. A single `ls` command can generate dozens of `stat` calls. Noise reduction collapses these into a smaller set of meaningful events.

<Callout type="tip" title="Noise reduction does not reduce security">
Noise reduction affects event reporting and dashboard display, not security evaluation. Every syscall is still evaluated by the proxy. Noise reduction only deduplicates the audit trail and dashboard feed.
</Callout>

## Creating a custom profile

Create a new profile by adding a section to your profiles file:

<ConfigExample
  title="~/.config/grith/supervisor/profiles.toml"
  code={`[data-science]
description = "Profile for data science notebooks and scripts"

routine_paths = [
    "~/data/",
    "~/notebooks/",
    "/tmp/",
]
denied_paths = [
    "~/.ssh/",
    "~/.aws/",
    "~/data/production/",    # Block access to production data
]

routine_commands = [
    "python",
    "pip",
    "jupyter",
    "conda",
    "git",
]

routine_destinations = [
    "pypi.org",
    "conda.anaconda.org",
]

network_policy = "score"

[data-science.noise_reduction]
enabled = true
deduplicate_reads = true
suppress_stat_calls = true`}
/>

Then use it:

<Terminal command='grith exec --profile data-science -- jupyter notebook' />

## Overriding pre-built profiles

You can override fields in a pre-built profile by redefining the section in your user config. User config values take precedence over shipped defaults.

For example, to restrict Claude Code's network access:

<ConfigExample
  title="~/.config/grith/supervisor/profiles.toml"
  code={`[claude-code]
# Override only the network policy; all other fields inherit from the default
network_policy = "deny"`}
/>

## Profile inheritance

If you want a custom profile that extends an existing one, use the `extends` field:

<ConfigExample
  title="~/.config/grith/supervisor/profiles.toml"
  code={`[claude-code-strict]
extends = "claude-code"
description = "Claude Code with stricter network and filesystem controls"

# Additional denied paths (merged with parent)
denied_paths = [
    "~/.env",
    "~/.npmrc",
    ".env",
    ".env.local",
]

# Override routine_destinations to empty (no score reductions for network)
routine_destinations = []

network_policy = "deny"`}
/>

<Terminal command='grith exec --profile claude-code-strict -- claude-code "refactor the auth module"' />

## Attaching to a running process

If an AI tool is already running, you can attach grith supervision to it by PID instead of spawning it:

<Terminal command="grith exec --profile generic --attach 12345" />

<Callout type="warning" title="Linux only">
Attaching to a running process requires ptrace permissions. On most Linux distributions this works for processes owned by the same user. On macOS, process-level attachment is limited to lifecycle events only.
</Callout>
