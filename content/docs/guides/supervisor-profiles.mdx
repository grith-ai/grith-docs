## Overview

When you run `grith exec`, the supervisor intercepts the tool's file, process, and network operations and evaluates them through the proxy. Supervisor profiles tell grith which of those operations are routine for a given tool.

Profiles are allowlist-shaped trust hints, not a second policy engine. Matching operations get low-score fast paths. Unusual operations still go through the full filter pipeline.

<Callout type="tip" title="New to profiles?">
If you want a step-by-step custom example, start with [Build Your First Supervisor Profile](/docs/guides/build-supervisor-profile).
</Callout>

## Where profiles live

The shipped profiles are currently loaded from:

<CodeBlock code={`config/supervisor/profiles.toml`} language="text" />

That file contains `[defaults]`, all built-in `[[profiles]]`, and the runtime overlay definitions. If you want to add or tune profiles in the current build, edit that file.

## Which profile to use

grith ships with 11 built-in profiles:

| Profile | Use it for |
|---|---|
| `generic` | Unknown or unclassified tools; strict fallback |
| `generic-cli` | Manual opt-in for generic developer CLIs that need GitHub / VS Code infrastructure |
| `grith-repl` | grith's built-in agent and REPL sessions |
| `claude-code` | Claude Code |
| `codex` | OpenAI Codex CLI |
| `aider` | Aider |
| `goose` | Goose |
| `copilot` | GitHub Copilot CLI |
| `cursor` | Cursor CLI |
| `cline` | Cline |
| `openclaw` | OpenClaw |

Use a built-in profile explicitly:

<Terminal command='grith exec --profile claude-code -- claude "fix the auth bug"' />

If you omit `--profile`, grith tries to auto-detect the tool from the command name and falls back to `supervisor.default_profile` if there is no known match.

## How a profile is resolved

Each session resolves to an effective policy:

1. `[defaults]` contributes universal local-only trust
2. the selected static profile resolves through `extends`
3. a provider overlay may apply for `grith-repl`
4. a launcher overlay may apply if the session was launched from a known IDE terminal

Launcher overlays are automatic. There is no `--launcher` flag on `grith exec`.

## Writing a custom profile

Profiles use the `[[profiles]]` TOML schema:

<ConfigExample
  title="config/supervisor/profiles.toml"
  code={`[[profiles]]
name = "my-tool"
display_name = "My Tool"
extends = "generic"
rationale = "My Tool edits project files, runs Python checks, and calls its API."

routine_paths = [
    "\${PROJECT_DIR}/**",
    "\${HOME}/.config/my-tool/**",
    "\${HOME}/.cache/my-tool/**",
    "/tmp/my-tool-*",
]

routine_commands = [
    "python", "python3", "pytest",
    "curl",
]

routine_destinations = [
    "api.my-tool.com",
    "pypi.org",
]

readonly_paths = [
    "\${HOME}/.config/my-tool/credentials.json",
]

readonly_path_patterns = []
routine_exec_roots = []
routine_listen_addresses = []`}
/>

## Picking a base

Most custom profiles should extend `generic`:

- `generic` is the strict fallback profile
- it trusts project-local paths only
- it inherits shared local defaults such as `git`, `bash`, `/proc`, and read-only SSH config access from `[defaults]`

Use `generic-cli` only when you are supervising a generic developer tool that truly needs the shared GitHub / VS Code network layer and does not already have a dedicated named profile.

## Field guide

### `routine_paths`

Add the directories and files the tool routinely reads and writes. Use `${PROJECT_DIR}` and `${HOME}` so the profile stays portable.

<Callout type="warning" title="Be conservative with routine_paths">
`routine_paths` trusts writes as well as reads. Do not use it for secrets or user config that should never be modified. If a path should be trusted for reads only, use `readonly_paths` or `readonly_path_patterns` instead.
</Callout>

### `routine_commands`

Add the executable basenames the tool normally spawns. Commands already covered by `[defaults]` do not need to be repeated.

### `routine_destinations`

Add hostnames the tool normally contacts. Matching is subdomain-aware, so `example.com` also covers `api.example.com`.

### `readonly_paths`

Use exact-match read-only trust for individual files such as config, known_hosts, or credentials that a tool may need to inspect but should never modify.

### `readonly_path_patterns`

Use narrow wildcard read-only trust for variable filenames such as `${HOME}/.ssh/*.pub`.

### `routine_exec_roots`

Use this only for trusted executable directories where the tool routinely spawns binaries outside normal `PATH` resolution.

### `routine_listen_addresses`

Use this for tools that are expected to bind local listener sockets, usually loopback addresses only.

### `launch_contract`

Some tools require specific launch arguments when supervised. Example:

<ConfigExample
  title="Launch contract"
  code={`launch_contract = { required_args = ["--sandbox", "disabled"] }`}
/>

grith enforces these contracts before launch.

## Launcher overlays

Launcher overlays add small amounts of context-specific trust when a session starts inside a known IDE terminal.

| Overlay | Detection | Current additions |
|---|---|---|
| `vscode-terminal` | Parent process is `code`, `code-insiders`, or `codium`; or `TERM_PROGRAM=vscode` | Adds `code` to routine commands |
| `cursor-terminal` | Parent process is `cursor` | No-op placeholder today |

They are defined under `[[launcher_overlays]]` and apply automatically. They do not replace the base profile.

## Provider overlays

Provider overlays add network destinations for `grith-repl` sessions:

| Overlay | Destinations |
|---|---|
| `openai` | OpenAI platform and auth endpoints |
| `anthropic` | Anthropic and Claude endpoints |
| `openrouter` | OpenRouter API |

Named external-tool profiles such as `claude-code` or `codex` already include their own provider destinations, so provider overlays are mainly for grith's built-in agent flows.

## Testing a profile

Run the tool under supervision and watch for queued or denied operations:

<Terminal command='grith exec --profile my-tool -- my-tool "run the task"' />

Then inspect the session logs:

<Terminal command="grith log --last-session" />

If a legitimate operation was queued:

- add the missing path to `routine_paths` or `readonly_paths`
- add the missing executable to `routine_commands`
- add the missing hostname to `routine_destinations`

If an operation feels too broad, remove it from the profile and let the normal filters handle it.

## Session allowlist debugging

The resolved effective policy becomes a flat session allowlist:

- `routine_paths` become recursive path allowlist entries
- `routine_commands` resolve to absolute executable paths
- `routine_destinations` become `net:` entries
- `readonly_paths` become `ro:` entries
- `routine_exec_roots` become `exec-prefix:` entries

That is why a small profile change can materially change live session behavior.

## Example: stricter clone instead of in-place override

Because profile names must be unique, the safest way to tighten a shipped profile is to create a new one that extends it:

<ConfigExample
  title="Derived stricter profile"
  code={`[[profiles]]
name = "claude-code-strict"
display_name = "Claude Code (Strict)"
extends = "claude-code"
rationale = "Claude Code with a reduced destination set for locked-down environments."

routine_paths = []
routine_commands = []
routine_destinations = [
    "anthropic.com",
    "api.anthropic.com",
]`}
/>

Use it with:

<Terminal command='grith exec --profile claude-code-strict -- claude "review src/auth.rs"' />
