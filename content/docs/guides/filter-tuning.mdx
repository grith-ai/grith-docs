## Overview

grith ships with sensible defaults, but every team and project has different risk tolerances. Filter tuning lets you control exactly how the security proxy evaluates tool calls — which filters run, how much weight each carries, and where the allow/queue/deny boundaries sit.

This guide covers threshold adjustment, per-filter configuration, dry-run testing, cold-start behaviour, and ready-made presets for common scenarios.

## Understanding the scoring model

Every tool call passes through the filter chain. Each filter produces an independent score. These scores are multiplied by their respective weights and summed into an aggregate score. That aggregate is compared against two thresholds:

| Threshold | Default | Effect |
|---|---|---|
| `auto_allow_threshold` | 3.0 | Scores below this are executed immediately |
| `auto_deny_threshold` | 8.0 | Scores above this are blocked outright |

Anything between the two thresholds lands in the digest queue for human review.

## Adjusting global thresholds

The simplest tuning is moving the threshold boundaries. If too many routine operations are being queued, raise the allow threshold. If dangerous operations are slipping through, lower the deny threshold.

<Terminal command="grith config set proxy.auto_allow_threshold 3.0" />

<Terminal command="grith config set proxy.auto_deny_threshold 7.0" />

Or edit the config file directly:

<ConfigExample
  title="~/.config/grith/config.toml"
  code={`[proxy]
auto_allow_threshold = 3.0
auto_deny_threshold = 7.0`}
/>

<Callout type="warning" title="Narrowing the queue zone">
Setting `auto_allow_threshold` too high means more operations execute without review. Setting `auto_deny_threshold` too low means more legitimate operations get blocked. Start with small adjustments (0.5 increments) and monitor the results with `grith audit`.
</Callout>

## Per-filter configuration

Each filter can be independently enabled, disabled, or weighted. Filter configs live under `[proxy.filters.<filter_id>]` in your config file.

### Disabling a filter

<ConfigExample
  title="~/.config/grith/config.toml"
  code={`[proxy.filters.reputation]
enabled = false`}
/>

### Adjusting filter weight

Weight is a multiplier applied to the filter's raw score before aggregation. The default weight for all filters is `1.0`. Increasing the weight makes a filter contribute more to the aggregate; decreasing it makes the filter less influential.

<ConfigExample
  title="~/.config/grith/config.toml"
  code={`[proxy.filters.secret_scan]
weight = 2.0    # Double the impact of secret detection

[proxy.filters.argument]
weight = 0.5    # Halve the impact of argument length checks`}
/>

### Filter-specific settings

Some filters accept additional configuration. For example, the `egress_policy` filter can be configured with allowed and denied destinations:

<ConfigExample
  title="~/.config/grith/config.toml"
  code={`[proxy.filters.egress_policy]
enabled = true
weight = 1.5
allowed_destinations = ["api.openai.com", "api.anthropic.com", "registry.npmjs.org"]
default_action = "score"    # "score" (default) or "deny"`}
/>

The `path_match` filter can be tuned with custom sensitive path patterns:

<ConfigExample
  title="~/.config/grith/config.toml"
  code={`[proxy.filters.path_match]
extra_sensitive_paths = [
    "/opt/secrets/",
    "/var/lib/myapp/credentials/",
]`}
/>

## Dry-run testing with grith proxy test

Before deploying filter changes, use `grith proxy test` to dry-run a tool call against your current configuration. The command evaluates the call through the full filter chain and reports the score without executing anything.

<Terminal command='grith proxy test fs.write /etc/passwd "root:x:0:0"' />

<CodeBlock
  code={`[proxy test] Dry-run: fs.write → /etc/passwd

  Filter results:
  ├─ operation_risk       +2.0  (write to system path)
  ├─ path_match           +3.0  (matches /etc/passwd)
  ├─ sensitive_path       +4.0  (critical system file)
  ├─ allowlist            +0.0
  ├─ argument             +0.0
  ├─ secret_scan          +0.0
  ├─ command              +0.0
  └─ aggregate score       9.0

  Decision: DENY (score 9.0 > threshold 8.0)
  Exit code: 2`}
  language="text"
/>

Exit codes from `grith proxy test`:

| Exit code | Decision |
|---|---|
| 0 | ALLOW |
| 1 | QUEUE |
| 2 | DENY |

This makes it straightforward to use in scripts:

<CodeBlock
  code={`#!/bin/bash
grith proxy test fs.write ./src/main.rs "fn main() {}"
case $? in
  0) echo "Would be allowed" ;;
  1) echo "Would be queued for review" ;;
  2) echo "Would be denied" ;;
esac`}
  language="bash"
/>

## Cold-start behaviour

During the first 200 tool calls in a new grith installation, the proxy operates in cold-start mode. The escalation zone widens to catch more edge cases while grith learns your baseline:

| Phase | Allow threshold | Deny threshold | Effect |
|---|---|---|---|
| Cold-start (first 200 calls) | 2.0 | 10.0 | Wider queue zone, more human review |
| Normal operation | Configured value | Configured value | Standard thresholds apply |

### Speeding up learning

If you are setting up grith on a project with predictable tool-call patterns, you can accelerate the cold-start phase:

**Run a representative workload.** Execute typical tasks that exercise the operations your agents will perform:

<Terminal command='grith run "build the project and run tests"' />

**Import an existing audit log.** If you have a previous grith installation, export and import its audit data to seed the adaptive model:

<Terminal command="grith audit export --format json > audit-export.json" />

**Manually end cold-start.** If you are confident in your filter configuration, you can disable cold-start behaviour entirely:

<ConfigExample
  title="~/.config/grith/config.toml"
  code={`[adaptive]
cold_start_calls = 0    # Skip cold-start phase entirely`}
/>

<Callout type="danger" title="Disabling cold-start reduces safety">
Setting `cold_start_calls = 0` means grith will use your configured thresholds from the very first call. If your thresholds are too permissive, dangerous operations may be auto-allowed before the adaptive model has any baseline data.
</Callout>

## Preset configurations

### Strict mode

Maximum security. Suitable for production environments or when working with untrusted models. Widens the queue zone so more operations require human review.

<ConfigExample
  title="strict-mode — ~/.config/grith/config.toml"
  code={`[proxy]
auto_allow_threshold = 1.5
auto_deny_threshold = 6.0

[proxy.filters.secret_scan]
weight = 2.0

[proxy.filters.egress_policy]
weight = 2.0

[proxy.filters.taint]
weight = 1.5

[adaptive]
cold_start_calls = 500    # Extended learning phase`}
/>

### Relaxed mode

For trusted environments with well-known models. Narrows the queue zone so fewer operations require manual review. Not recommended for production.

<ConfigExample
  title="relaxed-mode — ~/.config/grith/config.toml"
  code={`[proxy]
auto_allow_threshold = 4.0
auto_deny_threshold = 9.0

[proxy.filters.argument]
weight = 0.5

[proxy.filters.rate_limit]
weight = 0.5

[adaptive]
cold_start_calls = 100`}
/>

### Development mode

Optimised for local development. Allows most file and process operations within the project directory while still blocking clearly dangerous actions.

<ConfigExample
  title="dev-mode — ~/.config/grith/config.toml"
  code={`[proxy]
auto_allow_threshold = 3.5
auto_deny_threshold = 8.0

[proxy.filters.path_match]
weight = 0.5    # Reduce path sensitivity for project files

[proxy.filters.command]
weight = 0.75

[proxy.filters.egress_policy]
allowed_destinations = ["*"]    # Allow all outbound in dev
weight = 0.5

[adaptive]
cold_start_calls = 50`}
/>

<Callout type="info" title="Project-level overrides">
You can maintain different configs per project using the `--config` flag or by placing a `grith.toml` file in your project root. grith merges project-level config on top of the global config.
</Callout>

## Monitoring filter effectiveness

After making changes, monitor how your filters are performing:

<Terminal command="grith audit" />

Look for patterns: if a specific filter is consistently producing false positives (scoring safe operations too high) or false negatives (missing genuinely risky operations), adjust its weight or threshold accordingly.

The dashboard provides a visual breakdown of filter trigger rates:

<Terminal command="grith dashboard start" />

Navigate to the filter statistics panel at [http://localhost:3141](http://localhost:3141) to see per-filter hit rates, average scores, and score distributions over time.
