## Overview

grith includes a web dashboard that provides real-time visibility into agent activity, filter results, and session history. The dashboard is served by grith's built-in HTTP server and runs entirely on your local machine — no external services, no telemetry, no cloud dependency.

## Starting the dashboard

<Terminal command="grith dashboard start" />

<CodeBlock
  code={`[grith] Dashboard server starting...
[grith] ✓ Dashboard running at http://localhost:3141
[grith] Press Ctrl+C to stop, or use 'grith dashboard stop' from another terminal.`}
  language="text"
/>

Open [http://localhost:3141](http://localhost:3141) in your browser.

### Dashboard lifecycle

Check if the dashboard is running:

<Terminal command="grith dashboard status" />

Stop the dashboard:

<Terminal command="grith dashboard stop" />

The dashboard runs as a background process. It continues running after you close the terminal that started it. Use `grith dashboard stop` to shut it down.

## Dashboard architecture

The dashboard is a React frontend served by grith's Axum HTTP server. It communicates with grith via two channels:

| Channel | Protocol | Purpose |
|---|---|---|
| REST API | HTTP | Fetching audit logs, digest items, config, statistics |
| Live feed | WebSocket (`/ws/live`) | Real-time event stream for tool calls as they happen |

All data is read from grith's local SQLite database. The dashboard makes no external network calls.

<Callout type="info" title="Dashboard is optional">
The dashboard is a convenience layer. Everything it shows is also available via the CLI: `grith audit`, `grith digest review`, `grith supervisor list`. Use whichever interface fits your workflow.
</Callout>

## Features

### Real-time event stream

The main view shows tool calls flowing through the proxy in real time. Each event displays:

- **Timestamp** — When the tool call was intercepted
- **Tool call** — What operation was attempted (e.g., `fs.write → src/main.rs`)
- **Source** — Built-in agent (`grith run`) or supervisor session (`grith exec`)
- **Filter results** — Which filters fired, their individual scores
- **Aggregate score** — The combined score
- **Decision** — ALLOW, QUEUE, or DENY with colour coding

Events stream in via WebSocket with no polling. When a new tool call is evaluated, it appears in the feed within milliseconds.

### Digest review

The digest panel shows tool calls that scored in the queue zone and are waiting for human review. From the dashboard you can:

- **Approve** — Execute the queued operation
- **Deny** — Block the operation and discard it
- **Allow always** — Approve and add a rule so similar operations auto-allow in the future
- **Deny and terminate** — Block the operation and kill the session that produced it
- **Unlock egress** — Approve a network call that was blocked by containment mode

Each queued item shows the full filter breakdown, the tool call details, and the session context, giving you enough information to make an informed decision without digging through logs.

<Callout type="tip" title="Keyboard shortcuts">
The digest review panel supports keyboard shortcuts: `a` to approve, `d` to deny, `n` for next item, `p` for previous. These match the CLI digest review interface.
</Callout>

### Audit log browser

The audit panel provides a searchable, filterable view of all past tool calls. Filter by:

- **Time range** — Last hour, today, last 7 days, custom range
- **Decision** — ALLOW, QUEUE, DENY
- **Filter** — Show only events where a specific filter fired
- **Session** — Show events from a specific agent or supervisor session
- **Score range** — Show events within a score range

Click any audit entry to see the full detail: every filter's score, the raw tool call parameters, the session context, and the final decision with reasoning.

### Filter statistics

The statistics panel shows aggregate filter performance:

- **Trigger rates** — How often each filter fires (produces a non-zero score)
- **Average scores** — Mean score per filter over the selected time range
- **Score distributions** — Histogram of scores per filter
- **False positive indicators** — Filters that frequently score operations later approved by humans

Use this data to tune filter weights. If a filter has a high trigger rate but its flagged operations are consistently approved, consider lowering its weight. See [Filter Tuning](/docs/guides/filter-tuning) for guidance.

### Session monitoring

The sessions panel shows all active and recent sessions:

- **Active supervisor sessions** — Tools currently running under `grith exec`, with live event counts
- **Recent agent sessions** — Past `grith run` tasks with summary statistics
- **Per-session breakdown** — Tool call counts, allow/queue/deny ratios, duration, and active filters

Click an active session to switch the real-time feed to show only events from that session. This is useful when multiple supervised tools are running simultaneously.

### Supervisor session detail

For active supervisor sessions, the dashboard provides a dedicated view accessible at `/supervisor/:id` or by clicking a session in the sessions panel:

- **Live tool call feed** — Real-time events for this session only
- **Process tree** — The supervised process and its children
- **Containment status** — Whether the session is in containment mode and why
- **Session controls** — Kill the session, freeze/thaw the process

The supervisor detail view connects via a dedicated WebSocket at `/ws/supervisor/:id` for low-latency updates.

## Configuration

### Bind address

By default the dashboard binds to `localhost:3141`. To change the port:

<ConfigExample
  title="~/.config/grith/config.toml"
  code={`[server]
bind = "127.0.0.1:8080"`}
/>

<Callout type="warning" title="Binding to non-loopback addresses">
If you bind to `0.0.0.0` to allow network access, configure TLS or use a reverse proxy. See the [Reverse Proxy](/docs/guides/reverse-proxy) guide.
</Callout>

### Rate limiting

The server includes built-in rate limiting to prevent abuse:

<ConfigExample
  title="~/.config/grith/config.toml"
  code={`[server.rate_limit]
requests_per_second = 100
burst_size = 200`}
/>

## CLI equivalents

Every dashboard feature has a CLI equivalent:

| Dashboard feature | CLI equivalent |
|---|---|
| Real-time event stream | `grith log --tail` |
| Digest review | `grith digest review` |
| Audit log browser | `grith audit` |
| Audit export | `grith audit export --format json` |
| Session list | `grith supervisor list` |
| Session detail | `grith supervisor status <id>` |
| Kill session | `grith supervisor kill <id>` |
| Filter stats | `grith audit export --exfil-stats` |

## Troubleshooting

### Dashboard not loading

If the dashboard starts but the browser shows a blank page or connection error:

1. Verify the server is running: `grith dashboard status`
2. Check the port is not in use: `lsof -i :3141`
3. Check grith logs: `grith log --session dashboard`

### WebSocket disconnects

If the live event feed drops or shows a "disconnected" message:

- The dashboard automatically reconnects on WebSocket drop
- If behind a reverse proxy, ensure WebSocket upgrade headers are configured (see [Reverse Proxy](/docs/guides/reverse-proxy))
- Increase proxy read timeout if disconnects are frequent

### Slow performance with large audit logs

The audit browser loads data in pages. If the initial load is slow:

- Narrow the time range filter
- Use the decision filter to reduce the result set
- Consider increasing SQLite page cache in config:

<ConfigExample
  title="~/.config/grith/config.toml"
  code={`[general]
db_cache_size_mb = 64    # Default is 32`}
/>
