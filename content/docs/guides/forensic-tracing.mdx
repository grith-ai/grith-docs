## Overview

Forensic tracing captures every intercepted syscall **before** filtering decisions are applied, producing a JSONL file that serves as a complete audit trail of everything a supervised process attempted. This is distinct from grith's standard audit log, which records only calls that reach the scoring pipeline. The forensic trace includes syscalls that were noise-filtered, session-allowed, or auto-approved by reputation, giving you the full picture for post-incident analysis.

## Enabling forensic tracing

Pass the `--trace-syscalls-jsonl` flag to `grith exec`:

<Terminal command='grith exec --trace-syscalls-jsonl trace.jsonl -- claude-code "refactor auth module"' />

The trace file is written incrementally as syscalls are intercepted. Each line is a self-contained JSON object. If grith crashes or the process is killed, all previously written records are intact.

<Callout type="info" title="Performance impact">
Forensic tracing adds a write to the trace file for every intercepted syscall. For typical coding sessions this is negligible, but for syscall-heavy workloads (e.g., large builds), expect a small increase in latency. The trace file can grow to tens of megabytes for long sessions.
</Callout>

## Fields captured

Each JSONL record contains the following fields:

| Field | Type | Description |
|---|---|---|
| `event_id` | String | Unique identifier for this syscall event |
| `timestamp` | ISO 8601 | Time the syscall was intercepted |
| `session_id` | String | Supervisor session ID |
| `pid` | Integer | Process ID of the traced process |
| `tid` | Integer | Thread ID |
| `ppid` | Integer | Parent process ID |
| `stage` | String | Processing stage when this record was emitted (see below) |
| `decision` | String | Final decision: `allow`, `deny`, `queue`, or `pending` if not yet decided |
| `score` | Float | Composite proxy score (0.0 if not yet scored) |
| `raw_syscall_kind` | String | Low-level syscall classification (e.g., `FileRead`, `FileWrite`, `NetConnect`) |
| `tool_call_type` | String | Mapped tool call type used by the proxy (e.g., `fs.read`, `fs.write`, `net.connect`) |
| `exe_path` | String | Executable path of the process making the syscall |
| `argv` | Array | Command-line arguments of the process |
| `cwd` | String | Current working directory of the process |
| `comm` | String | Short process name (from `/proc/pid/comm`) |
| `env_selected` | Object | Selected environment variables: `PATH`, `HOME`, `SSH_AUTH_SOCK`, `GIT_SSH`, `GIT_SSH_COMMAND`, and any `CLAUDE_*` variables |

## Stage values

The `stage` field indicates where in grith's processing pipeline the syscall was when the record was written:

| Stage | Description |
|---|---|
| `captured` | Syscall intercepted, no processing yet |
| `noise_filtered` | Matched a noise reduction pattern (e.g., `futex`, `mmap`) and was auto-allowed |
| `session_allowed` | Matched a session allowlist entry from the active profile |
| `proxy_scored` | Passed through the full scoring pipeline |
| `denied` | Blocked by the proxy (score exceeded deny threshold or hard deny) |
| `approved` | Allowed by the proxy (score below allow threshold) |
| `reputation_auto_allow` | Auto-allowed by the adaptive reputation system (trust score >= 0.92) |

## Use cases

### Investigating agent startup behaviour

AI coding agents perform extensive filesystem and network activity during startup. Forensic traces let you see exactly which files Claude Code reads, which network connections it establishes, and which processes it spawns before accepting user input.

<Terminal command='grith exec --trace-syscalls-jsonl startup-trace.jsonl -- claude-code "exit"' />

Then filter for the first few seconds:

<CodeBlock
  code={`# Extract syscalls from the first 5 seconds
jq 'select(.timestamp < "2026-03-24T14:32:06Z")' startup-trace.jsonl | \
  jq -r '[.timestamp, .stage, .raw_syscall_kind, .exe_path] | @tsv'`}
  language="bash"
/>

### Tracing SSH key access chains

If an agent accessed SSH keys, the forensic trace reveals the full chain: which process read the key, what it did next, and whether any network activity followed.

<CodeBlock
  code={`# Find all SSH-related syscalls
jq 'select(.argv[]? | contains(".ssh"))' trace.jsonl`}
  language="bash"
/>

### Reconstructing exfiltration attempts

When a canary fires or a DLP block triggers, the forensic trace provides the full sequence of events leading up to the exfiltration attempt, including which files were read, how data was staged, and where the agent tried to send it.

<CodeBlock
  code={`# Extract the full event chain for a session with a deny
jq 'select(.session_id == "sup_a1b2c3d4")' trace.jsonl | \
  jq -r '[.timestamp, .stage, .decision, .tool_call_type, (.argv | join(" "))] | @tsv'`}
  language="bash"
/>

## Security considerations

<Callout type="danger" title="Trace files contain sensitive data">
Forensic trace files contain full command lines, file paths, and selected environment variables. They may reveal credential file locations, SSH socket paths, and internal directory structures. Store trace files with the same care as audit logs. Do not commit them to version control or share them over insecure channels.
</Callout>

Recommended practices:

- Store trace files in a directory with restricted permissions (`chmod 600`)
- Delete trace files after analysis is complete
- If sharing traces for team review, redact the `env_selected` field and any `argv` entries containing paths to credential files
- Use `grith audit export` for routine reporting; reserve forensic tracing for targeted investigations
