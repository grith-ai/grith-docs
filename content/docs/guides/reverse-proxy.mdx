## Why a reverse proxy

By default, grith's HTTP server binds to `localhost:3141`. This is secure for single-user use — only processes on the same machine can reach it.

When you need to expose the dashboard or API to other machines on your network (team access, remote monitoring, CI integration), you have two options:

1. **Native TLS** via `[server.tls]` in grith's config
2. **Reverse proxy** with TLS termination via nginx or Caddy

A reverse proxy is the better choice when you already have an nginx or Caddy instance managing certificates, need centralised access control, or want to add rate limiting and request logging at the gateway level.

<Callout type="danger" title="Never expose without TLS">
grith's API and dashboard transmit audit data, filter configurations, and session details. Exposing the server over plain HTTP on a network — even an internal one — allows anyone on that network to read and modify your security configuration. Always terminate TLS before the traffic reaches the network.
</Callout>

## Option 1: Native TLS

If you do not want to run a separate reverse proxy, grith supports TLS natively:

<ConfigExample
  title="~/.config/grith/config.toml"
  code={`[server]
bind = "0.0.0.0:3141"    # Bind to all interfaces

[server.tls]
cert_path = "/etc/grith/tls/cert.pem"
key_path = "/etc/grith/tls/key.pem"`}
/>

With native TLS enabled, grith serves HTTPS directly. This is the simplest option if you only need TLS and do not need the additional features of a reverse proxy.

<Callout type="info" title="Certificate management">
grith does not manage certificate renewal. Use a tool like certbot or a corporate CA to provision and renew certificates. grith reloads certificates on SIGHUP without requiring a restart.
</Callout>

## Option 2: nginx reverse proxy

### Prerequisites

- nginx installed and running
- TLS certificates (Let's Encrypt, corporate CA, or self-signed for internal use)
- grith server running on the default `localhost:3141`

### Configuration

<CodeBlock
  code={`# /etc/nginx/sites-available/grith

upstream grith_backend {
    server 127.0.0.1:3141;
}

server {
    listen 443 ssl http2;
    server_name grith.internal.example.com;

    ssl_certificate     /etc/letsencrypt/live/grith.internal.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/grith.internal.example.com/privkey.pem;

    # TLS hardening
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers on;
    ssl_session_cache shared:SSL:10m;

    # Security headers
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    add_header X-Content-Type-Options nosniff always;
    add_header X-Frame-Options DENY always;

    # API and dashboard
    location / {
        proxy_pass http://grith_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # WebSocket for live updates
    location /ws/ {
        proxy_pass http://grith_backend;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # WebSocket timeout — grith sends pings every 30s
        proxy_read_timeout 120s;
        proxy_send_timeout 120s;
    }

    # Redirect HTTP to HTTPS
    error_page 497 https://$host$request_uri;
}

server {
    listen 80;
    server_name grith.internal.example.com;
    return 301 https://$host$request_uri;
}`}
  language="nginx"
/>

Enable the site and reload nginx:

<Terminal command="sudo ln -s /etc/nginx/sites-available/grith /etc/nginx/sites-enabled/ && sudo nginx -t && sudo systemctl reload nginx" />

### Test the connection

<Terminal command="curl -s https://grith.internal.example.com/health | jq" />

<CodeBlock
  code={`{
  "status": "ok",
  "version": "1.0.0",
  "uptime_seconds": 3421

## Option 3: Caddy reverse proxy

Caddy handles TLS certificate provisioning automatically via Let's Encrypt or ZeroSSL. This is the simplest option if you do not already have an nginx infrastructure.

### Configuration

<CodeBlock
  code={`# /etc/caddy/Caddyfile

grith.internal.example.com {
    # TLS with automatic certificate management
    tls internal    # Use "internal" for self-signed, remove for Let's Encrypt

    # API and dashboard
    reverse_proxy localhost:3141

    # Security headers
    header {
        Strict-Transport-Security "max-age=31536000; includeSubDomains"
        X-Content-Type-Options nosniff
        X-Frame-Options DENY
    }
}`}
  language="caddyfile"
/>

<Callout type="tip" title="Caddy handles WebSocket automatically">
Caddy's `reverse_proxy` directive transparently handles WebSocket upgrade requests. No additional configuration is needed for grith's live event stream.
</Callout>

Reload Caddy:

<Terminal command="sudo systemctl reload caddy" />

### TLS options

| Caddyfile directive | Effect |
|---|---|
| `tls internal` | Self-signed certificate (suitable for internal networks) |
| `tls` (no argument) | Automatic Let's Encrypt certificate (requires public DNS) |
| `tls /path/to/cert.pem /path/to/key.pem` | Custom certificate from corporate CA |

## grith server configuration

When running behind a reverse proxy, keep grith bound to localhost. The proxy handles external-facing connections.

<ConfigExample
  title="~/.config/grith/config.toml"
  code={`[server]
bind = "127.0.0.1:3141"    # Only accept connections from localhost`}
/>

If grith detects it is bound to a non-loopback address without TLS configured, it logs a warning at startup:

<CodeBlock
  code={`[grith] WARNING: Server bound to 0.0.0.0:3141 without TLS.
[grith] This exposes the API and dashboard over plain HTTP.
[grith] Configure [server.tls] or use a TLS-terminating reverse proxy.`}
  language="text"
/>

## Access control

grith's HTTP server does not implement authentication on its own. When exposing via a reverse proxy, add access control at the proxy layer.

### Basic auth with nginx

<CodeBlock
  code={`# Add to the server block in /etc/nginx/sites-available/grith

location / {
    auth_basic "grith Dashboard";
    auth_basic_user_file /etc/nginx/.htpasswd;

    proxy_pass http://grith_backend;
    # ... other proxy headers
}`}
  language="nginx"
/>

Generate the password file:

<Terminal command="sudo htpasswd -c /etc/nginx/.htpasswd admin" />

### IP allowlisting with Caddy

<CodeBlock
  code={`grith.internal.example.com {
    tls internal

    @blocked not remote_ip 10.0.0.0/8 192.168.0.0/16
    respond @blocked 403

    reverse_proxy localhost:3141
}`}
  language="caddyfile"
/>

## WebSocket considerations

grith uses WebSocket connections for two features:

| Endpoint | Purpose |
|---|---|
| `/ws/live` | Real-time event stream for the dashboard |
| `/ws/supervisor/:id` | Live feed for a specific supervisor session |

Both require the reverse proxy to support WebSocket upgrade. The nginx configuration above includes the necessary `Upgrade` and `Connection` headers. Caddy handles this transparently.

If WebSocket connections drop frequently, increase the proxy read timeout. grith sends ping frames every 30 seconds, so a read timeout of 120 seconds provides adequate margin.

## Verifying the setup

After configuring the reverse proxy, verify all components work:

<CodeBlock
  code={`# Health check
curl -s https://grith.internal.example.com/health

# Dashboard (open in browser)
# https://grith.internal.example.com

# WebSocket (test with wscat)
wscat -c wss://grith.internal.example.com/ws/live`}
