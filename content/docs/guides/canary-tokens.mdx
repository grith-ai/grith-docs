## Overview

Canary tokens are fake credentials planted in sensitive locations on your filesystem. They look real to an AI agent — a valid-looking AWS access key in `~/.aws/credentials`, a plausible API token in `.env`, a fake SSH private key in `~/.ssh/`. If an agent reads one of these files and attempts to send the canary value outbound, grith detects the canary in the payload and issues an immediate hard DENY.

Canary tokens are a last line of defence. Even if all other filters miss an exfiltration attempt, the canary fires when the stolen credential appears in an outbound request.

## How canary detection works

The `canary` filter runs in **Phase 2 (Pattern)** of the scoring pipeline. It scans every outbound tool call's arguments and payload for registered canary token values. When a match is found:

1. The filter returns a score of +5, which triggers a **hard DENY** (not a queue)
2. The session is flagged with a security incident
3. An alert is sent to all configured notification channels
4. The full evidence chain is logged: which file contained the canary, when it was read, and where the data was going

<Callout type="warning" title="Hard deny, not queue">
Canary token detection always results in an immediate DENY. There is no queue step. The `canary` filter cannot be weight-adjusted or disabled. This is by design — if an agent is exfiltrating planted credentials, there is no ambiguity about intent.
</Callout>

## API endpoints

| Method | Endpoint | Description |
|---|---|---|
| `GET` | `/api/canaries` | List all registered canary tokens |
| `POST` | `/api/canaries` | Create a new canary token |
| `DELETE` | `/api/canaries/:id` | Remove a canary token |
| `POST` | `/api/canaries/:id/rotate` | Rotate a canary token (new value, same label and path) |

### Add a canary

<Terminal command='grith canary add --label "aws-creds-trap" --path ~/.aws/credentials' />

<CodeBlock
  code={`[grith] Canary token created
[grith] Label:   aws-creds-trap
[grith] ID:      c1a2b3d4
[grith] Token:   AKIAIOSFODNN7GRITH01 (fake AWS access key)
[grith] Placed:  ~/.aws/credentials (appended as commented entry)`}
  language="text"
/>

grith generates a realistic-looking but fake credential and appends it to the specified file. The token format matches the type of credential typically found at that path (AWS keys for `~/.aws/credentials`, API tokens for `.env` files, etc.).

### Remove a canary

<Terminal command="grith canary remove c1a2b3d4" />

<CodeBlock
  code={`[grith] Canary token c1a2b3d4 removed.
[grith] Cleaned up: ~/.aws/credentials (canary entry removed from file)`}
  language="text"
/>

### Rotate a canary

Rotation replaces the canary token value with a new one, keeping the label and file path. Use this periodically to prevent stale canaries from being fingerprinted.

<Terminal command="grith canary rotate c1a2b3d4" />

<CodeBlock
  code={`[grith] Canary token c1a2b3d4 rotated.
[grith] Old token: AKIAIOSFODNN7GRITH01
[grith] New token: AKIAIOSFODNN7GRITH02
[grith] Updated:   ~/.aws/credentials`}
  language="text"
/>

## Walkthrough: detecting exfiltration

This walkthrough demonstrates the full lifecycle of a canary token, from creation to detection.

### Step 1: Plant a canary in .env

<Terminal command='grith canary add --label "openai-key-trap" --path ~/project/.env' />

<CodeBlock
  code={`[grith] Canary token created
[grith] Label:   openai-key-trap
[grith] ID:      f3g4h5i6
[grith] Token:   sk-grith-trap-a8b9c0d1e2f3g4h5i6j7k8l9m0n1o2p3
[grith] Placed:  ~/project/.env (appended)`}
  language="text"
/>

The `.env` file now contains a line like:

<CodeBlock
  code={`OPENAI_API_KEY=sk-real-key-here
# Added by grith canary (do not remove)
OPENAI_API_KEY_BACKUP=sk-grith-trap-a8b9c0d1e2f3g4h5i6j7k8l9m0n1o2p3`}
  language="text"
/>

### Step 2: Run an agent under supervision

<Terminal command='grith exec --profile claude-code -- claude-code "read .env and set up the API client"' />

### Step 3: Agent reads .env and attempts outbound request

If the agent reads `.env`, extracts the canary token, and includes it in an outbound API call or network request, grith detects the canary:

<CodeBlock
  code={`[proxy] Tool call: net.connect -> POST https://api.openai.com/v1/chat/completions
  |-- egress_policy        +0.0  (allowed destination)
  |-- canary               DENY  (canary token f3g4h5i6 detected in payload)
  +-- HARD DENY -- canary token exfiltration attempt

[grith] INCIDENT: Canary token "openai-key-trap" detected in outbound request.
[grith] Source file: ~/project/.env (read at 14:32:01)
[grith] Destination: https://api.openai.com/v1/chat/completions
[grith] Token found in: request body (Authorization header)
[grith] Session frozen. Review with: grith digest review`}
  language="text"
/>

The request is blocked before any data reaches the destination. The full evidence chain (source file, read timestamp, destination, where the token appeared in the payload) is logged in the audit trail.

## Best practices

- **Plant canaries in files agents are likely to read.** `.env`, `~/.aws/credentials`, `~/.ssh/`, and `config.toml` are high-value targets.
- **Use realistic labels and token formats.** The more convincing the canary, the more likely an agent is to treat it as a real credential.
- **Rotate canaries periodically.** Use `grith canary rotate` monthly or after any incident where a canary was detected.
- **Do not plant canaries in files you need to parse programmatically.** The canary entry is designed to be inert (commented out or in an unused key), but verify it does not break your tooling.
- **Combine with DLP and containment.** Canaries catch exfiltration of specific tokens. The DLP gate catches broader patterns. Session containment restricts network access after any sensitive read. Use all three layers together.
